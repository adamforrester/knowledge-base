Prompt and Context InputThis cluster defines how user intent and context are captured and formatted before execution.Anatomy: At its core, the prompt input contains an auto-expanding multiline textarea, a prominent action button, character limits, context-attachment mechanisms (such as file or data chips), and slash-command popovers.Exemplars: Atlassian’s "Prompt Inline" and Amazon Cloudscape’s "Prompt Input" represent the industry benchmarks in this space.Convergence: The industry has converged on a rounded container layout, an inset submit button, and brief, descriptive placeholder text (e.g., "Ask a question") that does not truncate.Divergence: Systems diverge on how to handle context suggestions. Atlassian utilizes "Skills tags" (grey, rounded tags prefixed with a lightning bolt symbol) to restrict the prompt's execution scope. Cloudscape, on the other hand, utilizes "Support prompts" (selectable chips that can either populate the input field for editing or trigger execution immediately).Gth inline action utilities, status loading indicators, and retry controls.Exemplars: Amazon Cloudscape’s "Chat Bubble Stack" , Elisa's ChatArea , and Atlassian's "Chat Message Cards".Convergence: Design systems agree on partitioning complex markdown, tables, or list outputs into sequential "stacked" card containers to preserve layout integrity.Divergence: Visual loading indicators vary significantly. Cloudscape defines a structured "Thinking" pattern that displays the system's reasoning steps and the source documents it considered before generating the final text. Other systems simply display generic skeleton screens, pulsing icons, or linear progress bars.Trust, Transparency, and AttributionThis cluster focuses on validating machine-generated output to build user trust.Anatomy: Key elements include distinct visual badges, confidence ratings, inline citations, expandable source references, and standard refusal views.Exemplars: Carbon's "AI Label"  and Cloudscape's "Citation Popover".Convergence: The industry agrees that AI-generated content must be clearly labeled to ensure transparency.Divergence: Carbon strictly limits its AI Label (formerly the AI Slug) to non-actionable, explainability-focused popovers. It explicitly prohibits developers from mapping this label to action buttons like "regenerate". In contrast, Cloudscape focuses on factual verification, utilizing inline "Citation Popovers" and expandable "Sources" lists at the end of chat bubbles to help users verify facts.Feedback and CorrectionThis cluster provides the interface tools needed to capture user feedback for model refinement.Anatomy: Standard controls include persistent binary thumbs buttons, secondary feedback dialogs, and state restoration triggers.Exemplars: Cloudscape’s "Additional Feedback Dialog"  and Carbon’s "Revert" interaction model.Convergence: Binary feedback controls must be placed directly inside the message bubble and remain permanently visible on touch-screen devices without requiring hover interactions.Divergence: Carbtures a dynamic "Revert" pattern. If a user edits an AI-suggested input, the system removes the AI styling and replaces the label with an icon-only "Revert to AI" button, allowing the user to restore the initial model suggestion. Other systems treat generated text as static, offering no built-in way to recover the model's original output.Conversational Primitives and LayoutsThis cluster defines how the conversational workspace is structured on the screen.Anatomy: Layout primitives include slide-out sidebars, floating widgets, centered workspace overlays, thread cards, and status indicators.Exemplars: Atlassian's multi-layout Rovo integration  and GitHub’s Primer multi-agent guidelines.Convergence: Systems leverage collapsible sidebars and floating overlays to protect core workspace real estate on complex dashboards.Divergence: Multi-agent interaction models remain a point of divergence. GitHub's Primer provides detailed accessibility rules for announcing transitions between different agents to screen readers. However, commercial libraries continue to design exclusively for single-agent systems.Governance, Principles, and PoliciesThis cluster provides the guidelines and rules that sit alongside components to ensure safe, accessible, and compliant implementations.Anatomy: Key resources include human-in-the-loop policies, accessibility checklists, visual placement hierarchies, and content standards.Exemplars: Carbon for AI guidelines  and Primer Copilot Accessibility Principles.Convergence: Systems agree that AI-specific branding elements and styling should never be used purely for decoration.Divergence: Carbon provides prescriptive rules regarding component margins, sizes, and icon alignments to maintain a consistent visual language. Open-Source PrimitivesTo build standard chat features, teams can use several established open-source tools. The Vercel AI SDK provides hooks like useChat to manage message streams, loading variables, and error states. For the interface layer, Elisa offers a pre-built ChatArea component , Carbon Labs provides the @carbon-labs/ai-chat package , and third-party libraries like @katechat/ui offer components designed specifically for Mantine.To ensure accessibility, teams can use unstyled moe FrictionStatic design files struggle to model the dynamic, non-deterministic states of live AI interfaces. A designer cannot easily preview how a custom data table will transform when its AI-presence variable is toggled, nor can they simulate the visual transitions of streaming text, multi-step thinking processes, or dynamic error state recovery.To resolve this issue, Salesforce’s Lightning Design System 2 decouples visual styles from structural layouts. By prioritizing CSS Custom Properties (styling hooks) over hardcoded design tokens, Salesforce allows developer teams to write custom, agent-rendered layouts that adapt to dynamic data changes at runtime while remaining visually consistent.Machine-Readable Design SystemsAn interesting trend in design systems is the development of machine-readable interfaces. Libraries like Mantine and Elisa now publish official Model Context Protocol (MCP) server endpoints alongside standard documentation.Files like llms.txt and @mantine/mcp-server provide structured JSON-RPC metadata that allows AI coding tools (such as Cursor, Windsurf, and Claude Code) to read component APIs, props, and design guidelines directly. This represents a major shift: design systems are no longer built solely for human developers; they are now structured to allow AI agents to automatically assemble accessible interfaces.Architecture and governance — what shipping AI patterns actually requiresImplementing AI-aware design patterns successfully requires a robust data and governance foundation. Components-as-DataTraditional web applications render static pages using predefined component hierarchies. AI-powered applications, however, require components to be treated as dynamic data payloads. When a model determines that it should display a file preview, a financial table, or a dynamic action card, it does not output raw HTML. Instead, it generates structured JSON arguments representing a specific component signature.The application must parse these arguments and dynamically instantiate the correct design system component. If the design system is not built on a modular "components-as-data" architecture, the application will struggle to render these dynamic outputs, resulting in a generic, text-only chat experience.Structured MetadataTo enable AI coding agents and internal routing engines to navigate design assets, the system must expose its components as structured metadata. By implementing endpoints like the Model Context Protocol (MCP) server , developers can expose component properties, valid state configurations, and visual guidelines in a standard, machine-readable format. This metadata layer ensures that any automated front-end assembly complies with system guidelines, preserving accessibility and design intent.Evaluation Contracts in CI/CDIn traditional software development, design system compliance is verified through static visual tests and automated accessibility audits (e.g., checking contrast ratios and keyboard navigation in CI ). When deploying generative AI, however, the system's runtime outputs are non-deterministic, making static testing insufficient.To address this, design teams must establish evaluation contracts that run continuously in CI/CD pipelines. This includes deploying LLM-as-a-judge patterns to evaluate whether the application's runtime outputs match the appropriate design system components. For example, tests must verify that when a model outputs a list of references, the frontend correctly renders them inside a semantic sources expandable section rather than as unformatted body text.Similarly, evaluation suites must test refusal states to ensure that when a query violates safety guidelines, the system displays a standardized refusal view rather than returning raw API error codes.Content and Safety GovernanceThe design system defines the visual shell of an application, but it does not control the dynamically generated copy. This boundary is critical for accessibility and brand safety. As GitHub’s Primer outlines, a design system can provide a perfectly compliant, screen-reader-accessible chat container. However, if the underlying language model outputs unformatted blocks of text, uses jargon, or generates invalid HTML tags, it will break screen-reader compliance and increase cognitive load.To prevent this, the design system contract must extend to the model's system prompt instructions. Developers must enforce strict markdown guidelines at the model level, ensuring that outputs align with the layout, contrast, and cognitive guidelines established by the design system.Furthermore, these design systems must align with global regulatory disclosure frameworks. The European Union AI Act (specifically Article 52 regarding transparency obligations) mandates that users must be clearly informed when they are interacting with an AI system, unless this is obvious from the context of use. Similarly, emerging state-level disclosure laws in the United States demand clear and conspicuous labeling of machine-generated decisions and media.By standardizing primitives like Carbon's AI Label and explainability popover, design systems provide the essential visual indicators required to satisfy these legal obligations at scale.Failure modes — common ways AI-in-DS goes wrongIntegrating artificial intelligence into enterprise design systems introduces several unique operational, architectural, and behavioral failure modes.Assumed MaturityTeams often integrate experimental, lab-grade packages (such as Carbon Labs React components) directly into mission-critical production environments. This premature adoption leads to severe integration challenges. Because lab environments are designed for rapid prototyping rather than long-term stability, they lack stable API contracts, comprehensive cross-browser testing, and long-term maintenance plans.Visual Integration in a VacuumOrganizations frequently implement chat containers and prompt boxes as superficial visual layers without connecting them to robust asynchronous state engines. This decoupling leads to broken, unresponsive user interfaces. Without underlying logic to coordinate streaming buffers, manage loading states, and handle network interruptions, the front-end will display erratic rendering behaviors and fail to provide a reliable user experience.Lack of Technical AlignmentDesigners often create elaborate, high-fidelity mockups of conversational features (such as inline annotations and structured feedback inputs) without aligning with the engineering team's database capabilities. This visual-first approach results in non-functional features. If the backend cannot return structured token indexes, parse citation targets, or store granular, multi-attribute feedback data, the frontend remains a non-functional mockup.Compliance Violations (Disclosure Theatre)Some teams attempt to satisfy regulatory transparency guidelines by displaying static, non-interactive "AI" badges without providing genuine explainability pathways. This practice is legally and operationally risky. Under modern frameworks like the EU AI Act, a passive badge is insufficient if it does not explain how the data was processed. Carbon's AI Label avoids this issue by functioning strictly as a functional trigger for explainability popovers.Unnecessary Streaming (Streaming-as-Novelty)Teams often stream outputs word-by-word even when the system has pre-calculated the entire response (such as static system messages or brief database values). This over-use of streaming is counterproductive. Word-by-word streaming forces constant container resizing, increases CPU overhead, and slows down users who need to scan information quickly. Streaming should be reserved for long-form, non-deterministic content.Citation BlindnessDesigners frequently add citation superscripts or reference numbers to chat bubbles that are static, non-interactive, or point to broken links. This breaks the trust loop. A citation pattern is only effective if the user can interact with the marker to inspect the source material in context.ARIA Live Region SaturationA common accessibility mistake is configuring live screen-reader regions to announce every incoming token in a real-time stream. This creates an unusable auditory experience. Forcing a screen reader to announce every word as it renders blocks the user from executing other keyboard navigation commands. Instead, developers must configure quiet status updates or wait until a paragraph completes before notifying the user.Walled-Garden Custom Agent InterfacesIndividual product squads often build custom, ad-hoc conversational widgets that bypass the organization's central design system. This leads to a fractured user experience. These isolated implementations introduce inconsistent layouts, varying feedback icons, and non-standard visual styles across the product suite, eroding brand trust and increasing technical debt.Open questions and contested territoryAs the design systems field continues to evolve, several key challenges remain unresolved.                    ┌───────────────────────────────┐

Multi-Agent Interaction DesignWhile design systems have standardized single-agent chat interfaces, they have not yet established a visual language for multi-agent workflowges the traditional concept of a design system. Instead of serving as a static library of pre-built elements, the design system must function as a dynamic, generative runtime environment that can safely render layout changes on the fly.The Innovation LagMainstream design systems struggle to match the rapid release cycles of frontier consumer products (such as OpenAI Canvas, Gemini, or Claude Artifacts). These consumer platforms deploy new interaction models and split-screen layouts on a weekly basis.In contrast, enterprise design systems require months of research, accessibility auditing, and code refinement to promote a component to their stable core libraries. For digital agencies, this creates a significant strategic challenge:Bespoke Construction: Build custom, cutting-edge conversational interfaces that provide highly innovative features but carry high long-term maintenance costs and potential compliance risks.Conservative Adaptation: Rely exclusively on stable, pre-built design system components, which ensures accessibility and compliance but results in interfaces that lag behind the state-of-the-art.(a) Inaccessible and Non-AI SystemsOf the 18 target sources, SAP and the Power Apps Playbook were inaccessible due to restricted enterprise logins.
