# Popover — field-truth study (Claude pass)

## 1. Framing
A Popover is the **generic anchored transient surface** — a floating panel positioned relative to a trigger, dismissed by clicking outside or Escape, that does *not* trap focus. It is the third Overlay pole and the pivotal one, because it is where the **positioning substrate** that Select, Combobox, Menu, and (later) Tooltip have all been forward-referencing finally gets formalised. It stands on **Button** (the trigger — see button) and **Dialog** (the top layer + the `@starting-style`/`allow-discrete`/`overlay` exit-animation machinery — see dialog), and it is itself **the substrate those other components sit on**: Menu is a Popover whose content is `role=menu`; the Select/Combobox listbox is a Popover whose content is `role=listbox`; Tooltip is a (hint) Popover whose content is `role=tooltip`. Popover is the base; the others are specialisations distinguished by their content role and focus model.

**Two decisions define a from-scratch implementation:**
- **The native Popover API + CSS Anchor Positioning** (the platform-first decision, paired with Dialog's native-`<dialog>`): the **`popover` attribute** + `popovertarget` + `showPopover()`/`togglePopover()` give light-dismiss, Escape, top-layer, and one-open-per-ancestor for free (`popover="auto"`), while **CSS Anchor Positioning** (`anchor-name`/`position-anchor`, `anchor()`, `position-area`, `position-try`) gives declarative, collision-aware positioning that **replaces the Floating UI / Popper JS layer** the rest of the family leaned on. This is the most volatile platform claim in the catalogue — date it and note the verification path.
- **Light-dismiss + non-trapping** — the contrast with Dialog: a popover dismisses on outside-click/Escape, is *not* focus-trapped, and (when generic) does *not* auto-move focus in.

What it *isn't*: a **Tooltip** (a hover/focus *description*, `role=tooltip`, no interactive content, WCAG 1.4.13 — the sharpest boundary), a **Menu** (`role=menu` commands + roving — see menu), a **Dialog** (modal, focus-trapped — see dialog; though an interactive popover *is* a non-modal `role=dialog`, the overlap), or a **Drawer** (edge-anchored sheet). Why systems disagree: native-vs-Floating-UI, the focus model for interactive popovers, the trigger↔popover accessibility wiring, and where the popover/tooltip/menu/dialog lines fall.

## 2. Anatomy and parts
- **trigger / anchor** — the Button (or any element) the popover is positioned against and toggled from; carries `aria-expanded` + an accessible relationship to the surface (§6).
- **popover surface** — the floating panel (top layer); its `role` comes from content (§6).
- **arrow / caret** — an optional pointer connecting surface to anchor (positioned via CSS anchor positioning or a pseudo-element — §11); used for tight visual association, omitted for larger/looser panels.
- **content** — header/body/close as the content dictates (a generic popover has free-form content; a typed one — listbox/menu — has that structure).
- **close affordance** — an optional explicit close (icon Button) for interactive popovers; light-dismiss + Escape are the baseline.

## 3. Properties / API
- **`open` / `defaultOpen` + `onOpenChange`** — controlled/uncontrolled (fires on light-dismiss/Escape so external state syncs).
- **trigger/anchor composition** — `PopoverTrigger`/`asChild` (Radix forwards ARIA onto a real button); the anchor may differ from the trigger (anchor-to-one-element, trigger-from-another).
- **placement** — `side` (top/bottom/left/right) + `align` (start/center/end) + `sideOffset`/`alignOffset`; **collision** props (`avoidCollisions`/flip + shift, `collisionBoundary`/`collisionPadding`). Inherited positioning substrate (§11).
- **arrow** — `showArrow` + arrow size/element.
- **same-width** — match the trigger width (the Select/Combobox listbox case).
- **light-dismiss** — `closeOnEscape` (default true) + `closeOnInteractOutside`/`dismissable` (default true); `modal` (a *false* default — a popover is non-modal; `modal` raising it to trap+scrim shades into Dialog).
- **focus** — `initialFocus` (only when interactive — §6), `restoreFocus` (return to trigger, default true), **no `trapFocus`** (the Dialog difference).
- **role/aria** — `role` + `aria-labelledby`/`aria-describedby` per content; the trigger wiring (§6).

## 4. States and variants
- **States:** closed / open / **animating** (inherits Dialog's exit solution — §8).
- **Variants:**
  - **trigger:** click (default — interactive popover) vs **hover/focus** (`popover="hint"` / a rich tooltip — but interactive content must be click/focus, not hover-only, per 1.4.13).
  - **with-arrow / without.**
  - **generic vs role-typed** — info/content (generic, often `role=dialog` if interactive) / listbox (Select/Combobox) / menu (Menu) / tooltip (Tooltip). The role-typed ones are the specialisations.
  - **same-width-as-trigger** (listbox) vs **content-width** (generic).
  - **non-modal (default) vs modal** (the Dialog boundary).

## 5. Usage guidance
- **Use a Popover** for a small, transient, **contextual surface anchored to a trigger** that the user opens deliberately and dismisses by clicking away — supplementary info, a small form, a colour/emoji picker, a settings cluster, a profile card. It supplements the page without blocking it.
- **Don't use a Popover for:** a passive **description on hover/focus** (→ Tooltip — no interactive content, `role=tooltip`, 1.4.13), a **list of commands** (→ Menu — `role=menu` + roving), a **blocking decision** (→ Dialog — modal, focus-trapped), **edge-anchored secondary content/nav** (→ Drawer), or **critical/transient feedback** (→ Toast/Banner).
- **Decision frameworks:**
  - **Popover vs. Tooltip (the sharpest line):** interactive content + click-to-open (Popover) vs. a passive text description + hover/focus (Tooltip). A tooltip can't contain a link or button; the moment it does, it's a popover.
  - **Popover vs. Menu:** generic container (Popover) vs. a list of `menuitem` actions with roving tabindex (Menu). A popover *of links* is nav-in-a-popover, not a menu (the Menu rule).
  - **Popover vs. Dialog:** does it **supplement** (non-modal, light-dismiss, not trapped — Popover) or **block + demand resolution** (modal, trapped — Dialog)? An interactive popover is essentially a *non-modal dialog* (`role=dialog`), and the only real difference from Dialog is modality + the trap.
  - **When to move focus into a popover:** move focus in for an **interactive** popover (a form, a listbox — the user is going to operate it); do *not* for a hint/info popover (focus stays, the popover is supplementary). Always return focus to the trigger on close.

## 6. Accessibility
- **The native-popover accessibility GAP (the critical, non-obvious point):** the native Popover API wires *behaviour* (`popovertarget` toggles the surface) but **does NOT establish an accessible relationship** between trigger and popover. You must add it: **`aria-expanded`** on the trigger reflecting open state, and a semantic link — **`aria-details`** (for supplementary content the user can navigate to) or **`aria-controls`** + the appropriate **role** on the surface. `popovertarget` alone leaves a screen-reader user with a button that "does nothing" announced.
- **Role-by-content:** a popover has **no implicit ARIA role** — the role comes from what's inside: **`role="dialog"`** (interactive/non-modal — labelled by a heading), **`listbox`** (Select/Combobox), **`menu`** (Menu), **`tooltip`** (Tooltip). The generic interactive popover defaults to a **non-modal `role="dialog"`** with `aria-labelledby`. Don't slap `role=dialog` on a tooltip or a menu — match the content.
- **Light-dismiss + Escape** — `popover="auto"` gives click-outside + Escape dismissal natively; ensure both work and that dismissal returns focus appropriately.
- **The non-trapping focus model (the Dialog contrast):** a popover is **not focus-trapped** — Tab moves *through* it and *out* (into the rest of the page), unlike Dialog. Focus management:
  - **Generic/hint popover:** focus does **not** auto-move in; the surface is reachable in DOM order (or via the `aria-details` relationship).
  - **Interactive popover (form/listbox):** move focus to the first focusable / a designated element on open, and **return focus to the trigger** on close.
  - Because it's not trapped and not (by default) modal, the background stays interactive — which is the point.
- **WCAG 1.4.13 (content on hover or focus)** — for hover/focus-triggered popovers and tooltips: the content must be **dismissable** (Escape without moving the pointer), **hoverable** (the pointer can move onto it without it vanishing), and **persistent** (stays until dismissed/focus-lost). The recurring tooltip/hover-popover failure.
- **WCAG SCs:** 4.1.2 (name/role/value — the trigger relationship + content role), 1.4.13 (hover/focus content), 2.1.1 / 2.1.2 (keyboard, no trap — a popover must *not* trap), 2.4.3 (focus return), 2.4.7 (focus visible), plus the content's own SCs (listbox/menu/dialog).

## 7. Content guidelines
- **Concise, self-contained content** — a popover is supplementary; keep it scannable. A generic popover with a heading should label the surface via that heading (`aria-labelledby`).
- **The close affordance** — interactive popovers benefit from an explicit close (icon Button) beyond light-dismiss; hint/info popovers rely on light-dismiss/Escape.
- **The trigger** — the trigger's accessible name should convey what the popover holds ("Account menu", "Filters", "More info about X"); don't rely on the popover content alone.
- **Arrow** — use a caret/arrow when the visual association to the anchor would otherwise be ambiguous (a small popover near several controls); omit for large panels where placement is obvious.

## 8. Motion and transition
Origin-aware open/close — the surface scales/fades from the anchored edge (`transform-origin` at the trigger; ~120–180ms), exactly the Dialog substrate: `@starting-style` (from-state) + `transition-behavior: allow-discrete` (animate `display`/`overlay`) + the `overlay` property (stay in the top layer during exit) — inherited from Dialog, not re-derived. The arrow fades with the surface. Hover popovers respect a small open/close intent delay (the Menu safe-triangle lesson, where a submenu-like popover applies). Under `prefers-reduced-motion: reduce`, snap/opacity-only. (See dialog §8, 18-motion-foundations, 19-motion-implementation-web.)

## 9. Internationalization
- **RTL** flips placement and alignment (a `start`-aligned, right-side popover mirrors to the left; the arrow mirrors) — CSS anchor positioning with logical values (`position-area: inline-start`/`block-end`) handles this declaratively; the `position-try` fallbacks must be expressed logically so they mirror.
- **Text expansion** — popover content (especially forms/labels) expands; size to content within min/max bounds, no fixed widths.
- **Positioning** uses logical, not physical, axes so the collision fallbacks behave under RTL. (See 13-internationalization-and-rtl.)

## 10. Naming
The generic anchored surface is **`Popover`** (Radix, React Aria, Carbon, Polaris, Fluent, Ariakit, Base Web) or **`Popup`** (Atlassian); Primer frames the substrate as **`Overlay`** / **`AnchoredOverlay`** (the positioning primitive). **The practice default: `Popover`** as the generic anchored-surface primitive, with **Menu / Select / Combobox / Tooltip as the role-typed specialisations built on it** (not re-implementing positioning), the trigger via `PopoverTrigger`/`asChild`. Surface the positioning machinery as a shared primitive (an `Overlay`/`useAnchorPosition` layer) the whole family consumes. Aliases to map: `Popup`, `Overlay`, `AnchoredOverlay`, `FloatingPanel`, `Flyout`.

## 11. Implementation notes
- **Native Popover API + CSS Anchor Positioning first.** `popover="auto"` (light-dismiss + Escape + top layer + one-open-per-ancestor auto-close) / `popover="manual"` (no light-dismiss, no auto-close — you control it) / `popover="hint"` (the newer hover/hint layer that doesn't close other auto popovers). Pair with **CSS Anchor Positioning**: `anchor-name` on the trigger + `position-anchor` on the popover, `anchor()` / `position-area` for placement, and **`position-try` / `position-try-fallbacks` / `@position-try`** for declarative collision flip/shift — this is what **replaces Floating UI / Popper** for the whole overlay family.
- **What CSS anchor positioning doesn't yet cover (the Floating-UI fallback):** full support is uneven (the catalogue's most volatile platform claim — Chromium ahead, Safari/Firefox catching up; **verify before relying**), and some cases (complex arrow positioning, virtual/cursor anchors, scroll-following edge cases, older browsers) still want Floating UI. The practice default: **CSS-anchor-first, progressively enhanced, with Floating UI as the fallback/polyfill** during the transition.
- **The trigger↔popover accessible wiring** (the gap native leaves) — add `aria-expanded` to the trigger and `aria-details`/`aria-controls` + the content role; the headless libs do this, native doesn't (§6).
- **The focus model** — not trapped; move focus in only for interactive content; return to the trigger on close (§6).
- **The arrow** — position via CSS anchor positioning (anchor the arrow to the trigger too) or the legacy absolutely-positioned pseudo-element; keep it out of the accessibility tree.
- **Nested popovers + the light-dismiss stack** — `popover="auto"` auto-manages a stack (opening a non-descendant auto popover closes the previous; Escape closes the top), and the top layer stacks like Dialog; nested interactive popovers are allowed but watch the dismissal order.
- **The formalisation note:** this brief is where the positioning + light-dismiss substrate lives. Select, Combobox, and Menu's "positioning inherited from the overlay substrate" now resolves *here* — their listbox/menu surfaces are Popovers with a specific content role. Build the positioning once (native anchor positioning + a Floating-UI fallback) and have the family consume it.
- **Don't hand-roll** — Radix `Popover`, React Aria `Popover`/`usePopover`/overlay positioning, or Ariakit; they increasingly delegate to the native API + anchor positioning. Skin them.

## 12. Related and alternative components
- **Stands on:** Button (the trigger — see button), Dialog (the top-layer + `@starting-style`/`allow-discrete`/`overlay` animation substrate — see dialog), Icon (the close/arrow glyphs — see icon).
- **Is the substrate for (specialisations sitting on Popover):** Menu (`role=menu` content — see menu), the Select listbox (`role=listbox` — see select), the Combobox listbox (see combobox), Tooltip (`role=tooltip`, hint — see tooltip when briefed). These inherit Popover's anchored-surface + positioning machinery and add their content role + focus model.
- **Alternative to / boundary with:** Tooltip (passive description, hover/focus, no interactive content — the sharpest line), Menu (commands), Dialog (modal trap — but an interactive Popover *is* a non-modal `role=dialog`), Drawer (edge sheet — see drawer when briefed).

## 13. Field POV evolution
1. **The native Popover API arrived** — `popover`/`popovertarget`/`showPopover()` moved light-dismiss, the top layer, and one-open-per-ancestor into the platform, displacing the custom portal + outside-click-listener + manual stack.
2. **CSS Anchor Positioning is displacing Floating UI / Popper** — declarative `anchor()`/`position-area`/`position-try` is collapsing the JS positioning layer the whole overlay family depended on (the catalogue's most volatile, highest-impact platform shift — verify support).
3. **The top layer unified the overlay z-index story** (shared with Dialog) — popovers, dialogs, and their descendants stack predictably without z-index management.
4. **Role-by-content discipline hardened** — practice stopped treating "popover" as a role and started assigning `dialog`/`listbox`/`menu`/`tooltip` by content, and surfacing the native a11y gap (no automatic trigger relationship).
5. **Popover-as-base** — design systems increasingly build Menu/Select/Tooltip on one Popover/positioning primitive rather than re-implementing anchoring per component.
