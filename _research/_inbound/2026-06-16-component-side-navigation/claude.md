# Side Navigation — field-truth study (Claude pass)

## 1. Framing
Side Navigation is an app's **primary persistent wayfinding** — a vertical column of navigation Links, pinned to the inline-start edge, present across route changes, that tells the user where they can go and (via the active state) where they are. It closes the Navigation category and is the most composition-heavy brief in the catalogue: it **inherits Link** (the items + the `nav[aria-label]` landmark + `aria-current="page"` contract — see link, breadcrumbs), **Drawer** (its mobile/collapsed-overlay form — see drawer), **Accordion/Disclosure** (collapsible sub-sections — see accordion), and **Tooltip** (the collapsed-rail icon labels — see tooltip). Its own substance is the *persistence + responsive transform*, the *collapsed-rail-vs-expanded* model, and the *multi-level active-state* contract.

**The defining decision is persistent-structure vs. transient drawer + the responsive transform:** a side nav is a **persistent part of the page layout** (in the grid/flex shell, not an overlay) — and on narrow viewports it **transforms into a modal Drawer behind a hamburger**. This is exactly the boundary Drawer set up from the other side (a drawer is the side nav's temporary/mobile form).

What it *isn't*: a **Drawer** (transient overlay — the side nav is the persistent structure; the drawer is its mobile form — see drawer), a **Menu** (`role=menu` *commands*; side nav is *navigation links* — the nav-is-not-a-menu rule, see menu), **Tabs** (in-page panel switching, not cross-page navigation — see tabs), **Breadcrumbs** (a secondary location *trail*, not primary nav — see breadcrumbs), or a **Tree** (`role=tree` for hierarchical *data*; site nav is nav+disclosure+links — §6). Why systems disagree: the rail-vs-expanded default, the multi-level reveal (in-place disclosure vs flyout), and Ant's overloading of "Menu" for what is really side nav.

## 2. Anatomy and parts
- **nav landmark** — `<nav aria-label="…">` (a *unique* label so it's distinguishable from the header/breadcrumb/footer navs — §6), wrapping lists of items.
- **header** — the product/logo area at the top (often the collapse toggle's home).
- **item** — a **Link** (`<a href>`): **icon + label** + an **active indicator** ("you are here") + optional **badge/count**; the collapsed rail shows icon-only (+ tooltip).
- **collapsible group / section** — a section header + nested items, expanded in place via the **Disclosure** pattern (`button[aria-expanded]` + region) — *not* `role=menu`.
- **collapse / expand toggle** — a `button[aria-expanded]` that switches expanded ↔ rail.
- **footer** — account/settings/user pinned at the bottom.
- **rail tooltip** — the collapsed-rail item's label exposed on hover/focus (inherited Tooltip).

## 3. Properties / API
- **`items` / `children`** — the nav tree (a data model `{label, href, icon, items?}` or composed `NavItem`/`NavGroup` children); each item inherits Link's `asChild` for the framework router.
- **`collapsed` / `defaultCollapsed` + `onCollapsedChange`** — the rail toggle; **persist the choice** across sessions (§11).
- **active/current item** — the component marks the matching item `aria-current="page"` (often derived from the current route).
- **multi-level** — nested `items` + per-group expanded state (`defaultExpanded`/controlled), rendered as in-place disclosure (expanded) or a flyout Popover (collapsed rail).
- **`renderItem` / `asChild`** — the Link-vs-router slot (inherited Link).
- **mobile breakpoint / `as="drawer"`** — below the breakpoint, render as a modal Drawer behind a hamburger (inherited Drawer).

## 4. States and variants
- **Item states:** rest / hover / focus-visible / **active** (`aria-current="page"`, the "you are here" indicator — distinct from hover/focus) / disabled.
- **Variants:**
  - **expanded** (icon + label) vs **collapsed rail** (icon-only + tooltip; sub-items via flyout).
  - **single-level** vs **multi-level** (nested disclosure groups).
  - **persistent** (desktop, in the layout) vs **mobile drawer** (modal, behind a hamburger).
  - **with-icons / with-badges**; the **section group** (grouped + labelled).
  - (Material's responsive taxonomy: **navigation rail** (compact, icon rail) / **navigation drawer** (expanded) / **navigation bar** (bottom, mobile) — the same nav at different breakpoints.)

## 5. Usage guidance
- **Use Side Navigation** as **primary persistent wayfinding** for an app with a **broad or deep IA** — dashboards, admin/SaaS, consoles, docs — where the user moves between many top-level sections and benefits from them being always present.
- **Don't use it for:** a **shallow or marketing site** (→ a top nav / header bar), **in-page panel switching** (→ Tabs — not navigation), a **location trail** (→ Breadcrumbs — secondary), a **transient panel** (→ Drawer — though the mobile side nav *is* a drawer), or **hierarchical data** (→ Tree — `role=tree`).
- **Decision frameworks:**
  - **Side nav vs. top nav:** many sections / deep IA / app-shell (side) vs. few sections / marketing / content site (top header bar). Many apps use both (top bar + side nav).
  - **Side nav vs. Tabs:** cross-page navigation that changes the route (side nav — links + `aria-current`) vs. in-page view switching (Tabs — `tablist`, panels). The routing-is-not-a-tablist rule (see tabs).
  - **Rail vs. expanded:** expanded when labels aid scanning / there's room; collapse to a rail to reclaim space (icon-only **must** carry tooltips/labels — §6). Persist the user's choice.
  - **Persistent vs. drawer:** persistent in the layout on desktop; modal drawer behind a hamburger on mobile — the responsive transform.

## 6. Accessibility
- **The nav landmark + unique label:** `<nav aria-label="Main">` (or `aria-labelledby`) — a *unique* name so a screen-reader landmark list distinguishes it from the global header nav, breadcrumb nav, and footer nav (the Breadcrumbs/Link rule reused). Items are a `<ul>`/`<li>` list of Links.
- **`aria-current="page"`** on the active item (inherited from Link — the "you are here" signal; a visual `.active` class alone is insufficient). In the multi-level case, the active child is `aria-current="page"` and its ancestor group is expanded/visually indicated (not itself `aria-current`).
- **The Disclosure pattern for sub-sections, NOT `role=menu`:** collapsible groups use a `button[aria-expanded]` revealing a region/list of Links — the same heading+button+region pattern as Accordion. **`role=menu`/`menuitem` is wrong for site navigation** (the nav-is-not-a-menu correction — see menu); Ant's "Menu"-for-nav is the documented anti-pattern.
- **The keyboard model is TABBED, not roving:** a side nav is a *list of links* — Tab moves through them in order; it is **not** a roving-tabindex widget (not a menu, menubar, or tree). The collapse toggle is a `button[aria-expanded]`; disclosure group headers are buttons.
- **The collapsed-rail label requirement:** an icon-only rail item still needs an **accessible name** (visually-hidden label or `aria-label`) *and* a visible **Tooltip** on hover/focus (inherited Tooltip) — an icon alone is not a name.
- **`role=tree` only for a true data tree** — a hierarchical *data* structure (a file explorer) is `role=tree` with roving tabindex; *site navigation* is nav+disclosure+links. Don't reach for treeview for a nav.
- **The skip link** — a "Skip to main content" Link (inherited Link) lets keyboard users bypass a long nav.
- **WCAG SCs:** 4.1.2 (name/role/value — the landmark + `aria-current` + the toggle), 1.3.1 (the list/landmark structure), 2.4.1 (bypass blocks — skip link), 2.4.5 (multiple ways), 2.4.8 (location — the active state), plus Link's SCs (2.4.4 link purpose) and the collapsed-rail Tooltip's 1.4.13.

## 7. Content guidelines
- **Concise, scannable item labels** — a word or two, noun-led, matching the destination's page title; front-load the distinguishing word.
- **Icon discipline** — icons aid recognition but **must pair with a text label** (or a tooltip in the rail); never icon-only without a name (see icon's icon-only trap).
- **Section grouping** — group related destinations under a labelled section header; keep groups shallow (deep nesting is an IA smell — prefer a flatter structure or a sub-page).
- **The active label** + the **collapsed-rail label** (the tooltip text = the item label). **Badges/counts** are concise ("3", "99+") with an accessible name ("3 unread").

## 8. Motion and transition
- **Collapse/expand** animates the nav **width** (rail ↔ expanded) — labels fade/clip as the width changes; a quick ease (~150–200ms). (Width/`grid-template-columns` animation; avoid layout thrash.)
- **Sub-item reveal** — in-place disclosure animates the region height (inherits the Accordion/`::details-content`/`calc-size` approach — see accordion); the collapsed-rail **flyout** is a Popover (origin-aware — see popover).
- **The mobile drawer** slides from the edge (inherits Drawer's slide + `@starting-style` — see drawer).
- Under `prefers-reduced-motion: reduce`, snap the width/height and drop the slide. (See 18-motion-foundations, 19-motion-implementation-web.)

## 9. Internationalization
- **RTL** — the nav flips to the **inline-end** edge via logical properties (`inset-inline-start` → mirrors), the collapse-toggle chevron and any sub-item chevrons mirror, and the mobile drawer slides from the mirrored edge (inherited Drawer/logical-placement). Don't hardcode `left`.
- **Text expansion** — nav labels expand (+30–50%); the expanded nav must accommodate longer labels (a wider column or wrapping) without truncating to ambiguity; the rail width is icon-driven and stable. (See 13-internationalization-and-rtl.)

## 10. Naming
**`Side Navigation`** / **`SideNav`** (Carbon) and **`Sidebar`** (shadcn — the collapsible-rail reference) are the common names; the item-list primitive is **`NavList`** (Primer) or **`Navigation`** (Polaris); Material frames it as the **navigation rail / navigation drawer**; Fluent and Ant call the vertical case **`Nav`** / **`Menu`** (Ant's overload — flagged). **The practice default: `SideNavigation`** (or `Sidebar`) for the persistent shell, composing **`NavItem`** (a Link) / **`NavGroup`** (a Disclosure of items) / the collapse toggle, with the mobile form delegated to **`Drawer`** and the rail labels to **`Tooltip`**. Aliases to map: `Sidebar`, `SideNav`, `Nav`, `NavList`, `Navigation`, `NavigationRail`, `NavigationDrawer`.

## 11. Implementation notes
- **The persistent layout shell** — the side nav is a column in the app's **grid/flex shell** (`grid-template-columns: auto 1fr`), not a `position:fixed` overlay; collapsing it animates the column width and reflows the main content. State (collapsed/expanded) **persists** across sessions (localStorage/cookie/user-pref).
- **The mobile-drawer transform** — below the breakpoint, render the *same nav content* as a modal **Drawer** behind a hamburger (inherits Drawer — focus-trap, scrim, the slide; see drawer). Share the item model between the persistent and drawer renders.
- **The collapsed rail + flyout** — collapsed shows icons + tooltips; sub-items that were in-place disclosures become **flyout Popovers** anchored to the rail item (inherited Popover positioning). Hovering/focusing a rail item reveals its label (Tooltip) and, if it has children, its flyout.
- **Sub-nav is Disclosure, not `role=menu`** — `button[aria-expanded]` + a region of Links; reuse the Accordion mechanics (`<details>`/`::details-content` where it fits — see accordion). Never `role=menu` for site nav.
- **`aria-current` + router integration** — derive the active item from the current route and set `aria-current="page"`; each item renders the framework router's Link via `asChild` while emitting a real `<a href>` (inherited Link).
- **Don't hand-roll** — shadcn `Sidebar`, Primer `NavList`, Carbon `SideNav`, or compose Link + Accordion + Drawer + Tooltip; skin them.

## 12. Related and alternative components
- **Stands on:** Link (the items + the `nav[aria-label]` landmark + `aria-current` contract + `asChild` router — see link, breadcrumbs), Drawer (the mobile/collapsed-overlay form — see drawer), Accordion/Disclosure (collapsible sub-sections — see accordion), Tooltip (collapsed-rail icon labels — see tooltip), Icon (item icons — see icon).
- **Alternative to / boundary with:** the **top nav / app header** (shallow/marketing vs deep-IA persistent side nav — not yet briefed), **Tabs** (in-page panel switching, not cross-page nav — see tabs), **Breadcrumbs** (a secondary location trail, not primary nav — see breadcrumbs), **Drawer** (transient overlay; the side nav *is* a drawer on mobile — see drawer), **Menu** (`role=menu` commands, not nav links — see menu; the nav-is-not-a-menu boundary), **Tree** (`role=tree` for hierarchical data, not site nav).

## 13. Field POV evolution
1. **The collapsible-rail Sidebar standardised** — shadcn's `Sidebar` (and Material's navigation rail / Carbon's SideNav) made the icon-rail-collapse + persisted-state + flyout pattern the expected app-shell nav.
2. **The nav-is-not-a-menu correction reached side nav** — multi-level nav settled on the Disclosure pattern (button+region+links), and `role=menu`-for-nav (Ant's overload) is the documented anti-pattern.
3. **The responsive side-nav ↔ drawer transform converged** — the same nav content renders as a persistent column on desktop and a modal drawer behind a hamburger on mobile.
4. **`aria-current="page"`** standardised as the active-item signal (over a visual class alone), aligned with Link.
5. **The tree-only-for-data-tree clarification** — site navigation stopped (mis)using `role=tree`/treeview, reserving it for true hierarchical data.
