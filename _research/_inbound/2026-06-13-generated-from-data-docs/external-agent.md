Generated-From-Data Documentation: Architecture and Commercial Default Strategy1. The Architecture in ProductionThe transition from manually authored prose to generated-from-data documentation represents a fundamental architectural shift in design system engineering. By 2026, the architecture that treats the component data file as the absolute source of truth has matured from a theoretical ideal into a robust, deployable reality. This model derives human-readable documentation, AI-readable metadata, Figma library properties, and rigorous conformance evidence from a singular, centralized substrate. Establishing this as a commercial default requires deep comprehension of the exact shapes, targets, and pipelines that constitute this ecosystem in production.The Component-Data-File ShapeThe foundational decision in this architecture is the shape and syntax of the component data file. The field currently navigates a tension between implicit, framework-native definitions and explicit, framework-agnostic data schemas.Framework-native definitions leverage the existing implementation layer as the primary substrate. For React environments, this typically involves react-docgen-typescript extracting types, interfaces, JSDoc comments, and default values directly from the source code. For Svelte-based systems, tooling like sveld analyzes the abstract syntax tree to emit structured JSON detailing props, slots, and reactive bindings, identifying conservative sources of emitted types without evaluating arbitrary code. The primary advantage of this implicit approach is exceptionally low friction; the developer authors TypeScript or Zod schemas as they normally would, and the documentation pipeline infers the rest. However, this closely couples the system's metadata to its runtime framework, presenting significant structural challenges for multi-platform systems.Conversely, explicit definitions utilize a neutral format—such as JSON Schema or YAML—to model the component's anatomy, properties, states, and design tokens independent of any specific platform implementation. This "components as data" paradigm dictates that a UI element's schema defines its intent, variants, and dependencies before a single line of code is written. While this approach requires a heavier upfront investment in custom parsers and build tooling, it allows a single schema to project out to web components, native iOS frameworks, and AI metadata registries simultaneously. Adobe Spectrum's architecture, for instance, utilizes extensive JSON schema definitions to map component APIs, track deprecations, and align tokens across varying platform outputs.Schema ArchitecturePrimary Extraction MechanismDownstream EnablersDownstream LimitationsFramework-NativeTypeScript / Zod / sveld ASTImmediate integration with Storybook Autodocs; requires minimal extra syntax for web-only teams.Locks metadata to the code execution context; struggles with cross-platform representation; vulnerable to complex inheritance masking.Explicit DataAgnostic JSON Schema / YAMLUnifies s-platform implementations; highly readable for AI and infrastructure tooling; enforces strict domain modeling.Requires custom intermediate compilation steps to generate runtime code and prop tables; introduces a secondary syntax layer.The Generation TargetsA mature generated-from-data pipeline projects the core source of truth to five distinct consumption targets, each requiring a specific projection of the underlying data.Storybook Autodocs: The standard, interactive delivery surface for developers. The pipeline dynamically generates the controls panel, prop tables, and interactive variants directly from the source types, allowing developers to manipulate component states in real-time without writing boilerplate stories.MDX Docs Site: The comprehensive portal for the broader product team. It pulls in type signatures, default values, and deprecation flags, dynamically merging them with long-form editorial content.The .ai.json and DESIGN.md Registries: The delivery surface for AI agents. The data file is translated into heavily structured metadata, often utilizing the W3C Design Tokens Format or the DESIGN.md specification, enabling coding agents to access canonical schemas, spatial relationships, and context-aware behavioral rules.Figma Library Properties: Utilizing technologies like Figma Code Connect, the source code's property definitions are mapped to Figma component variants, surfacing true-to-production code snippets directly in Dev Mode.Conformance Dossiers: The pipeline exports raw automated accessibility testing data directly into structured Accessibility Conformance Reports (ACRs) or Voluntary Product Accessibility Templates (VPATs), providing continuous, auditable compliance evidence.Where these projections diverge is in their required altitude of detail and formatting. A human reading the MDX site needs visual examples, spatial reasoning, and nuanced rationale; an AI agent consuming the .ai.json requires strict typing, semantic boundaries, and highly deterministic variant logic optimized for context windows ; Figma Code Connect requires literal framework-specific node mappings.The CI SurfaceThe continuous integration (CI) pipeline acts as the rigid enforcement mechanism for the generated architecture. The freshness check is the critical barrier ensuring that no generated artifact diverges from its source. If a developer alters a property in the TypeScript file but fails to commit the corresponding update to the generated schemas, the CI build fails, blocking the merge.Beyond simple freshness, the CI surface extends to multifaceted validation. Schema validation ensures that the explicitly defined JSON/YAML strictly adheres to the system's overarching metamodel. Accessibility schema enforcement runs tools like axe-core in headless environments against all generated component variants, identifying missing ARIA attributes or contrast failures, and blocking merges that introduce Web Content Accessibility Guidelines (WCAG) violations. Visual regression tools, such as Chromatic, capture automated screenshots of the generated stories to verify visual stability and detect unintended CSS cascading effects. Furthermore, semver bump detection automatically propagates deprecation notices across all five generation targets simultaneously whenever an API signature changes, ensuring that a deprecated property in code is immediately flagged in Figma and the MDX documentation.The Single-Source-of-Truth File StructureTo execute this architecture, the file structure within the repository is heavily compartmentalized. This strict compartmentalization ensures that machine-generated facts are never commingled with human-authored intent in a way that risks destructive overwrites during the build process. A standard component directory resembles the following cluster:Button.tsx: The runtime implementation..types.ts / schema.json: The authoritative API definition..stories.tsx: The programmatic permutations for visual testing..docs.mdx: The overlay file where generated data meets human prose.DESIGN.md / .ai.json: The aggregated AI metadata export, combining tokens and schema constraints..a11y.json: The automated compliance and keyboard navigation mappings resulting from CI testing.The Authoring SeamCrucially, generation does not entirely replace authoring; it elevates it. The discipline of the architecture lies in deeply understanding the "seam" between what a machine can accurately compute and what a human must explicitly explain.The generated pipeline owns all technical facts: prop types, default values, required flags, token mappings, inheritance chains, and DOM structures. The human author retains absolute ownership over intent. Authored prose is exclusively reserved for the "Purpose," "When to Use," "Avoid When," "Content Guidelines," and nuanced accessibility interactions (such as specific screen reader announcements contingent on complex application state).Attempts to generate intent via Large Language Models (LLMs) consistently result in failure. When an LLM attempts to author "purpose" prose without human editorial input, the resulting documentation is dangerously generic, frequently hallucinating business rules or entirely missing the subtle constraints of the specific enterprise domain. Therefore, the MDX overlay file acts as the designated authoring seam, utilizing import statements to pull the generated facts while preserving a protected, CI-verified space for expert editorial prose. The discipline is ensuring that authored prose remains authored, establishing a clear boundary of trust within the system.2. The Pipeline Shape: The Concrete Dependency GraphTo ship generated-from-data documentation commercially, practitioners must architect and deploy a rigid, deterministic pipeline. The dependency graph dictates exactly what runs when, what parallelizes, what blocks downstream execution, and how the system recovers from failures.From Source Change to Consumer UpdateThe pipeline triggers upon a commit to the centralized repository. The initial stage involves static extraction. For React-based systems, tooling parses the abstract syntax tree of the component source, extracting interfaces and JSDoc annotations. If this step encounters a malformed type or an unresolved inheritance chain, the build fails immediately, preventing cascading errors throughout the downstream documentation.Following successful extraction, the pipeline forks into multiple parallel generation paths, ensuring that all consumers receive synchronous updates.The Storybook and MDX PathIn the interactive documentation path, the extracted metadata object is piped directly into the Storybook engine. The react-docgen-typescript parser maps the TypeScript interfaces into Storybook's argTypes, automatically populating the controls panel without manual story configuration. Concurrently, the addon-a11y plugin executes axe-core against the rendered component DOM in the background , while Chromatic triggers parallel visual regression testing to capture state variations.The MDX docs site path operates as a sophisticated aggregator. The MDX compiler ingests the component's markdown file, but at render time, it dynamically resolves custom documentation components (e.g., <PropsTable component="Button" />) by querying the centralized data object. This guarantees that the technical specifications on the docs site are dynamically injected at build time, ensuring they are never more than one CI cycle removed from the production reality. Cross-references between components are validated by link checkers to ensure no dead documentation paths exist.The AI Metadata Emission PathThe .ai.json and Model Context Protocol (MCP) emission path requires a highly specialized compilation step optimized for machine readability. The pipeline extracts the technical schemas, token definitions, and any custom JSDoc @ai tags, merging them with the authored "when-to-use" rules extracted from the MDX file.This merging process must be aggressively deterministic. As demonstrated by enterprise AI Metadata Systems (AIMS), zero-contamination guarantees are achieved through independent MDX parsers that segment the data into discrete semantic categories (behavior, state, layout). The consolidation pipeline explicitly avoids using LLMs during the merge to prevent hallucinated data fabrication.Once merged, the data is formatted. Industry benchmarking has proven that semantic chunking (concept-aligned boundaries) achieves a 92% retrieval accuracy for LLMs, vastly outperforming monolithic JSON dumps or raw Markdown, while simultaneously reducing AI token consumption by up to 88%. This semantically chunked metadata is then served via an MCP server. When an AI agent—such as Atlassian's Rovo Dev CLI—queries the system using tools like ads_plan, the MCP server delivers the exact import paths, token applications, and accessibility requirements needed to generate compliant code.The Figma Library Sync PathThe pipeline interacts with Figma design assets through Code Connect. A local Code Connect CLI execution runs within the CI environment, scanning the repository for .figma.ts, .figma.tsx, or .figma.swift mapping files. It parses these templates, verifying that the code-side properties (e.g., a boolean isDisabled or an enum ButtonVariant) map correctly to the defined Figma component properties.This architecture currently operates largely as a unidirectional sync (Code projecting into Figma Dev Mode), surfacing the exact component API and framework-specific snippets to designers and developers inspecting the canvas. If a property is deleted in the codebase but remains present in the Figma design library, the CLI outputs a warning during the build, alerting the design systems team to the architectural drift.The Conformance Dossier PathThe final branch of the generation pipeline produces the conformance dossier. The CI extracts the raw, automated testing results from the axe-core headless runs and merges them with serialized logs of manual assistive-technology audits (which are stored as data alongside the components). This aggregated data is mapped section-by-section to specific Web Content Accessibility Guidelines (WCAG) and Section 508 criteria. The output is a highly structured Accessibility Conformance Report (ACR) that reflects the exact, verifiable state of the current system deployment, eliminating the need for periodic, reverse-engineered compliance audits.Pipeline Failure ModesIn 2026, the primary failure mode practitioners encounter involves complex monorepo resolutions. When utilizing tools like Vite, Nx, and Turborepo, generators frequently fail to extract inherited properties if the component is imported from a workspace package root rather than the direct source file. This results in empty prop tables or type definitions resolving simply to string instead of specific literals. The system recovers by either enforcing strict source-import linting rules across the monorepo or by requiring practitioners to manually redeclare critical interface extensions within the component file. If the metadata extraction silently fails, the pipeline drift goes unnoticed until the AI agent hallucinates an API during code generation, emphasizing the absolute necessity of strict schema validation checks before the CI allows a merge.3. The Hours-and-Cost ShiftTransitioning a design system engagement to a generated-from-data default dramatically alters the commercial mathematics of a project. It shifts the economic model from a high ongoing operational expenditure (Opex) defined by manual maintenance, to a higher upfront capital expenditure (Capex) focused on infrastructure, ultimately reducing the lifetime cost of ownership and significantly improving system reliability.The Upfront Engineering InvestmentScaffolding the necessary generation infrastructure is not trivial; it requires highly specialized engineering. In 2026, establishing a commercial-grade pipeline requires configuring the extraction tooling, building the custom MDX rendering templates, defining the .ai.json schema architecture, writing the CI freshness scripts, tuning the Code Connect CLI mappings, and standing up the foundational Model Context Protocol (MCP) server.For a standard React-based, single-platform system, this upfront investment realistically requires 60 to 80 engineering hours. For cross-platform systems requiring explicit JSON schemas mapped to multiple frameworks (e.g., Web, iOS SwiftUI, and Android Jetpack Compose), the investment scales rapidly to 120–160 hours. This cost encompasses the nuanced configuration of Turborepo workspaces to prevent docgen failures , the orchestration of deterministic data merging for the AI endpoints , and the establishment of the automated accessibility dossier pipeline.The Per-Component ShiftHistorically, documenting a single component via authored prose required between 3 to 8 hours of dedicated labor. This was typically split between a UX designer writing behavioral guidelines and an engineer manually typing out prop tables, formatting code examples in Markdown, capturing state screenshots, and manually verifying cross-references.Under the generated-from-data model, the per-component documentation burden shifts drastically. Because the prop tables, type signatures, default values, deprecation flags, and baseline interactive examples are programmatically derived from the source code, the engineering documentation time drops to near zero. The UX designer's time is subsequently refocused entirely on high-value, authored intent (the "seam" of purpose, accessibility context, and specific business logic). Assuming a conservative pipeline setup cost of 80 hours and a conservative per-component savings of 3 hours, the pipeline mathematically pays for itself at the 27th component. For a standard enterprise design system containing 40 to 60 foundational components, the generated architecture achieves profitability prior to the v1.0 release.However, the true commercial benefit materializes over the 18 to 36 month post-handoff window. In traditionally authored systems, documentation silently drifts. Code implementations evolve, APIs are deprecated, but the Markdown files remain static. The cost of rectifying this technical debt during an annual audit, or the catastrophic cost of compliance failures due to outdated VPATs , vastly outweighs the initial pipeline investment. Generated documentation ensures continuous alignment, reducing maintenance overhead by an estimated 70% over a three-year lifecycle, while guaranteeing that AI coding agents always possess the correct API surface.Commercial Discipline in the SOWTo execute this commercially, the practice must exercise extreme discipline in how it structures the Statement of Work (SOW). The practice must itemize the pipeline as a distinct, billable deliverable. Treating generation infrastructure merely as a sub-bullet under "Documentation" obscures its value, minimizes the engineering complexity, and forces the agency to absorb the setup hours as unbillable overhead.The pipeline must be positioned as a "Foundations-tier" architectural deliverable—analogous to CI/CD setup, design token architecture, or deployment environments. It is the infrastructure that makes the system usable at scale.4. The Decision Framework: Justifying Generation vs. AuthoringDespite the overwhelming long-term advantages, generated-from-data documentation is not a universal necessity for every engagement. The practice employs a stringent decision framework during the proposal-shaping conversation to determine whether the client's context justifies the heavy engineering investment.The Contextual VariablesThe decision hinges on several interconnected variables that dictate the complexity and lifespan of the system:Component Count Threshold: The baseline heuristic is the "30+ components" rule. Systems smaller than this rarely achieve the break-even point to justify the pipeline, making authored prose sufficient unless other variables dominate.System Lifetime Expectation: A rapid 6-month build-and-handoff for an ephemeral marketing site does not suffer from long-term drift; generation is overkill. A multi-year retainer or a long-term enterprise SaaS platform demands generation to survive.Consumer-Product Count: A design system serving a single product team can often survive on institutional memory and basic Storybook configurations. A system serving twelve distributed product teams requires determiniso-drift metadata to function without overwhelming the core support team.Multi-Platform and Multi-Brand Reach: A web-only, single-brand system is relatively simple. A system supporting five distinct brands across Web, iOS, and Android requires explicit data schemas to maintain parity. Authored prose in a multi-platform scenario guarantees fragmentation, contradiction, and ultimate failure.Regulatory Constraints: Clients subject to strict regulatory environments (European Accessibility Act, Section 508, HIPAA) require continuous, auditable accessibility conformance. Manual generation of VPATs/ACRs is highly error-prone and rapidly outdated; the necessity of automated dossier generation justifies the pipeline independently.AI-Readiness Procurement: Increasingly, enterprise clients mandate that their systems integrate seamlessly with AI coding agents (Cursor, Claude, Rovo, Windsurf). If AI-readiness is on the procurement checklist, generation is mandatory; agents cannot reliably ingest unstructured, authored prose and require an MCP or structured DESIGN.md integration to prevent hallucination.The 5-Question Decision TreeDuring the proposal phase, the practice applies the following diagnostic framework to guide the client:Scale: Will the system exceed 30 components and serve more than two distinct consuming product teams? (If Yes, lean toward Generation).Longevity: Is the anticipated lifecycle of the system greater than 18 months? (If Yes, lean toward Generation).Complexity: Must the system enforce parity across multiple platforms (Web/Native) or multiple brand themes? (If Yes, mandate Generation).Compliance: Is there a strict legal or contractual requirement for continuous Accessibility Conformance Reports (ACRs)? (If Yes, mandate Generation).AI-Integration: Does the client's engineering culture utilize or plan to utilize AI coding agents that require programmatic context? (If Yes, mandate Generation).If three or more answers are "Yes," the practice formally proposes the generated-from-data pipeline as the commercial default. If fewer than three apply, the practice scopes standard authored prose, explicitly outlining the manual drift risks in the proposal to protect the agency against future liability.5. Migration Paths: Transitioning to a Generated SubstrateFor clients with existing, mature design systems, the transition from an authored-prose environment (Notion, Confluence, GitBook, Zeroheight, or a hand-rolled site) to a generated substrate requires a highly disciplined migration playbook. The most critical realization is that treating migration as purely an engineering task is an anti-pattern; it is fundamentally a data-remediation and editorial process.The Two-Phase Migration PlaybookThe practice executes migrations over a 6 to 12-week timeline, divided into two distinct phases to minimize disruption to consuming teams.Phase 1: Parallel Scaffolding and Extraction.The engineering team stands up the extraction pipeline, configuring react-docgen-typescript, sveld, or the explicit JSON schemas alongside the existing codebase in a non-blocking branch. Simultaneously, the design team performs a rigorous editorial audit of the legacy documentation. The explicit goal is to separate immutable facts from editorial intent. Stale authored prose attempting to describe props, broken code snippets, deprecated static screenshots, and manually formatted API tables are systematically marked for deletion.What is meticulously preserved is the editorial intent: the purpose statements, "when-to-use" guidelines, accessibility context, and tone-of-voice instructions. This legacy intent is mapped into the new MDX overlay files or structured within the markdown body of the new DESIGN.md files.Phase 2: Dual-Publishing and Cutover.The generated pipeline is activated, publishing the new artifact (Storybook Autodocs + MDX site + AI metadata) alongside the legacy site. This dual-publish window (typically 2 to 4 weeks) allows consuming teams to update their workflows, verify API representations, and report any extraction anomalies. Once the generated site reaches feature parity regarding editorial content, the old site is permanently retired.The Hosted Design-Surface Reality (Zeroheight)A common friction point in migrations is the client's desire to retain hosted, design-friendly surfaces like Zeroheight. While the generated architecture treats the codebase component data file as the absolute source of truth, platforms like Zeroheight are traditionally designed as primary authoring environments.The migration path resolves this tension by inverting the relationship: Zeroheight becomes a read-only consumer surface. The practice utilizes the Zeroheight REST API to synchronize component statuses, markdown content, and design tokens from the codebase directly into the platform. While this satisfies design stakeholders who prefer visual interfaces, practitioners must acknowledge the failure modes: API sync cadences can break, and Zeroheight's programmatic content ingestion remains constraining compared to a native MDX pipeline. When the sync fails, the code-side generated site must be established as the definitive arbiter of truth.Migration Anti-PatternsThe most destructive anti-pattern during migration is discarding all existing documentation simply because it was hand-authored, effectively burning years of hard-won institutional knowledge regarding component usage constraints. Conversely, allowing legacy, manually written prop descriptions to persist in the new MDX files defeats the entire purpose of the architecture, leading to immediate and confusing contradictions with the source code's generated output. Finally, migrating the documentation without simultaneously setting up the CI freshness checks ensures the new system will begin rotting the day it is deployed.6. Within the practice's architectural positioning, it is merely one pillar of a broader cluster of deliverables that form the "AI-Readiness Commercial Default."The Four-Prerequisite Minimum ArchitectureTo achieve true AI-compatibility—where an agent can reliably reason about, implement, and audit a design system—an architecture requires four interrelated prerequisites. Generated documentation acts as the central, human-facing delivery mechanism for this cluster:Tokens with Descriptions: Semantic design tokens exported via the W3C Design Tokens Community Group (DTCG) format.Component Metadata: The explicitly generated .ai.json registry or the newly standardized DESIGN.md specification.Agent Instructions: Repositories require explicit instructional layers—such as AGENTS.md and CLAUDE.md—to route the AI's behavior, enforce coding standards, and dictate file structures.The Figma Pipeline: Unifying variables to Style Dictionary to Code Connect, ensuring the visual asset is programmatically linked to the code asset.Altitude of the Same ArtifactThe critical insight for practitioners is that the generated documentation site and the AI-readable metadata are not separate products requiring separate maintenance streams; they are the exact same artifact rendered at different altitudes.The same CI process that builds the interactive MDX props table builds the DESIGN.md YAML frontmatter  and outputs the semantically chunked JSON payload for the MCP server. When the Atlassian Rovo Dev CLI queries the Atlassian Design System (ADS) MCP server for button usage using the ads_plan tool, it receives the exact same API constraints, token mappings, and accessibility guidelines that a human reads on the documentation website. The generated documentation is simply the human-optimized delivery surface for the underlying four-prerequisite architecture.Commercializing the Unique IPCurrently, critical aspects of this cluster remain proprietary practice IP. Furthermore, while Figma Code Connect is publicly available, setting it up as a comprehensive bridge that actively informs an MCP server is a complex integration  that the practice possesses the specific capability to execute.The proposal strategy, therefore, must sell the entire cluster. Selling a "docs site" commoditizes the work and invites price comparison with basic offshore development. Anti-Patterns and Tooling Realities in 2026To operate as true experts, practitioners must acknowledge where the field's theoretical capabilities exceed its current tooling maturity. Pretending the generative ecosystem is flawless leads to disastrous project estimations and broken trust. In 2026, several significant gaps require specific practitioner workarounds.The Limits of Autodocs and LLM IntentA pervasive anti-pattern is the assumption that "Storybook Autodocs alone is enough." Autodocs excels at extracting physical prop types and basic JSDoc comments, but it fundamentally lacks the capacity to capture editorial intent, business logic, or complex behavioral patterns.Attempting to bridge this gap by commanding an LLM to "generate everything including intent" is equally flawed. When AI attempts to author "purpose" or "when-to-use" prose without human editorial input, the resulting documentation is dangerously generic and frequently contradicts the subtle business rules of the specific enterprise. For instance, an LLM might state a red button is for "destructive actions," missing that in a healthcare system, red specifically denotes "critical patient alert." Human curation at the authoring seam remains non-negotiable.Monorepo Tooling FailuresThe tooling chain for TypeScript extraction remains frustratingly fragile in complex environments. Specifically, react-docgen-typescript frequently breaks in Turborepo, Nx, or Vite workspaces when dealing with cross-package inheritance. If a component extends an interface from a different package root, the generator often outputs empty prop tables or defaults to obscure, unhelpful object types. Practitioners must work around this by enforcing strict direct-source import paths or by writing custom Vite plugins to manually resolve the AST, adding unforeseen hours to the pipeline setup.The Complex State CliffAutomated visual generation hits a hard wall at the "complex multi-variable state cliff." While generating a standard primary button is trivial, programmatic visual rendering algorithms struggle to reliably capture deeply nested, multi-variable states (e.g., a data table cell that is simultaneously selected, focused, and displaying a server-error flag). Automated attempts often result in broken UI renders. The practitioner workaround relies on explicitly defined, hardcoded story variants for these edge cases, rather than trusting the pipeline to generate the permutations automatically.Code Connect Asymmetry and Figma SyncWhile Figma Code Connect is maturing, a stark platform asymmetry exists. In 2026, Code Connect for React and React Native is largely baseline-grade, easily integrated via template files. However, support for iOS (SwiftUI), Android (Jetpack Compose), and HTML Web Components—despite recent official support additions —remains materially less mature regarding complex variant mapping and dynamic property resolution. Cross-platform systems inevitably collide with this asymmetry, requiring manual intervention and complex figmaApply helpers to maintain sync in Swift or Kotlin. Furthermore, the broader Figma sync seam remains problematic; design-side libraries still do not natively pull their absolute truth from the code repository automatically, allowing drift to creep in as design files are updated manually by designers on separate cadences.The MCP Server and Metadata RealityFinally, the field does not yet possess a turnkey, off-the-shelf "Design System MCP server" that an agency can simply install. Building a custom MCP server that securely parses design tokens, validates component metadata, manages semantic chunking strategies , and handles the JSON-RPC lifecycle  remains a highly bespoke, 40-to-80-hour engineering project.Furthermore, while Google's DESIGN.md specification is rapidly standardizing the structure of design metadata for AI , the internal .ai.json schema utilized by many practices remains proprietary IP. Practitioners must be prepared to build, maintain, and defend this infrastructure manually until vendor platforms fully internalize the architecture. This reality underscores the value of the practice's commercial offering: clients are not buying a theoretical spec; they are buying the proven engineering capability to make the architecture function in production today.
