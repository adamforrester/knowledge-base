Segmented ControlIdentityName: Segmented ControlAliases: Connected Button Group (Material 3), ToggleButtonGroup (MUI), ContentSwitcher (IBM Carbon), ActionGroup (Adobe Spectrum), ButtonGroup (Base Web).Inherits: components/toggle-button.md, components/radio.mdThe Segmented Control represents a high-density, horizontally adjoined group of atomic options that allow users to select from a mutually exclusive or collectively exhaustive set of choices. Functionally, the Segmented Control serves as a macro-container for the atomic Toggle Button component. It inherits its exact interaction states, visual resting properties, and pressed-state semantics from the individual toggle button. Logically, for the single-select variant, the component inherits the mutual exclusivity, data-binding architecture, and keyboard navigation model of the standard Radio Button group.However, the Segmented Control is not merely a reskinned radio group or a styled cluster of buttons. It is a distinct architectural pattern defined by its horizontal spatial constraint, its immediate temporal execution, and its unique visual composition. Establishing an opinionated default for this component requires confirming how it inherits from its foundational components while simultaneously defining the strict boundaries that separate it from other selection paradigms. The single-select variant shares the identical semantic structure of a radio group—enforcing a roving tabindex where only one active element is present at any given time, and where selection follows focus. Yet, it discards the radio button's vertical rhythm and its deferred-action form submission model. The Segmented Control is built for instantaneous view manipulation, requiring an entirely distinct set of accessibility and implementation strategies.DescriptionThe Segmented Control is arguably the most semantically contested component in modern design system architecture. While its physical presentation is broadly recognized across the industry—a contiguous horizontal row of discrete options—its functional boundaries overlap heavily with radio groups, standard tabs, and native select menus. Establishing an opinionated default for this component requires deeply deconstructing these heavily confused boundaries. Systems must determine precisely when a practitioner should reach for a Segmented Control over its functional siblings to avoid cognitive dissonance and layout failure.The Segmented-vs-Radio BoundaryThe semantic architecture of a single-select Segmented Control is identical to that of a radio group; both allow users to select exactly one option from a mutually exclusive list. However, the boundary between the two is rigidly defined by presentation density, spatial orientation, and the temporal nature of the selection action. Radio buttons are historically utilized within standard form data collection. They are optimized for vertical scanning, feature small touch targets (the radio input itself), and fundamentally imply a deferred action—the user makes a selection, but the system state is not mutated until a form submission occurs.Conversely, the Segmented Control implies immediate application. When a user selects a segment, the user interface must respond instantaneously. The component is optimized for horizontal constraint and high-density user interfaces where vertical space is at a premium and where large, easily tappable regions are required (adhering to Fitts's Law). Consequently, a Segmented Control should be reserved exclusively for instantaneous view filtering, sorting modifications, or inline data mutations. The radio group remains the absolute standard for declarative, multi-step data collection. If a selection requires a distinct "Save" or "Apply" button to take effect, the Segmented Control is the incorrect component, and a standard Radio Group must be utilized.The Segmented-vs-Tabs BoundaryThe distinction between Segmented Controls and Tabs is the most frequently violated boundary in digital product design. Both components feature horizontal, single-select arrays that update the view. The architectural fork relies entirely on the conceptual hierarchy and structural depth of the interface.Tabs are a disclosure and navigation pattern. Architecturally, they utilize role="tablist" and switch entirely distinct <tabpanel> elements. Tabs follow the physical metaphor of a filing cabinet—two tabs contain entirely distinct sheets of paper with divergent layouts, differing interactive elements, and separate data fetching life-cycles.A Segmented Control, however, does not switch panels; it manipulates the existing panel. It functions as a value, filter, or view switcher. For example, IBM Carbon (v10/v11) explicitly designed its "ContentSwitcher" to toggle alternate formatting or filtering of the same underlying dataset (e.g., switching a list of data from a "Grid" view to a "Table" view, or switching an inbox from "All" to "Unread"). Furthermore, GitHub Primer explicitly forbids utilizing a Segmented Control to change URLs or act as standard site navigation, reserving such behavior exclusively for tabbed interfaces or navigational links. If the selection completely unmounts the current DOM structure to render a vastly different interface, Tabs are required. If the selection merely re-sorts, filters, or changes the visualization of the current data context, the Segmented Control is the mathematically correct pattern.The Segmented-vs-Select BoundaryThe boundary between a Segmented Control and a Dropdown Select is defined by cognitive load, physical interaction cost, and strict horizontal real estate constraints. A Segmented Control exposes all available options immediately, maximizing "information scent." It requires a single physical tap or click to change state, optimizing for rapid, repeated interactions. However, it consumes significant horizontal real estate, scaling linearly with the number of options.A Select menu collapses these options. It saves horizontal real estate but introduces a significant interaction tax: one tap to open the menu, a cognitive and spatial search through the list, and a second tap to execute the selection. The opinionated default mandates that a Segmented Control should be utilized when there are between two and five critical options that the user must toggle frequently. This 2-5 segment "sweet spot" balances immediate visibility against the constraints of mobile viewports. If the option count exceeds five text-based segments, the interface must strictly fall back to a Select menu or an ActionMenu to prevent horizontal overflow, text truncation, and tap-target violations. The only exception is for icon-only segments, which may safely extend to six items due to their predictable, compact dimensional footprint.AnatomyThe physical construction of the Segmented Control has diverged into two distinct, heavily debated visual paradigms over the past decade. These paradigms are largely influenced by the underlying operating system design languages: the Track-and-Pill model (derived fundamentally from Apple's Human Interface Guidelines) and the Border-Collapse model (derived from Google's Material Design system).The Track-and-Pill ModelThe canonical iOS UISegmentedControl (updated significantly in iOS 13 and persisting through June 2023 guidelines) defines the component as a unified master container (the track) equipped with internal padding, housing a freely floating visual indicator (the pill) that rests beneath the active text or icon. The overarching track is typically rendered with a subdued, low-contrast background (often incorporating an inner shadow or slight darkening to establish physical depth), while the active pill is elevated with a drop shadow, a contrasting background, and rounded corners that mirror the track's outer radius.In this anatomy, vertical dividers are placed strictly between unselected segments to delineate the touch targets. The logic governing these dividers is highly dynamic: a divider must disappear if the segment to its immediate left or right becomes selected, as the elevated pill naturally establishes the visual boundary. This model emphasizes the concept of a single, continuous physical switch that the user slides back and forth.The Border-Collapse Model (Material 3 Connected Button Group)Google's Material 3 ecosystem explicitly deprecated the older "Segmented Button" in its Expressive Update (documented for late 2024/2026 platform states), shifting the entire ecosystem to the "Connected Button Group". Unlike the iOS model, the Connected Button Group entirely abandons the concept of a master track and a sliding pill. Instead, it relies on a series of independent toggle buttons joined seamlessly at their physical borders.The structural integrity of this model is maintained by highly specific border-radius manipulation. The outermost edges of the first and last buttons retain standard rounding (e.g., fully rounded pill shapes or specific border radii based on size variants), while the internal intersecting borders are strictly squared off (border-radius of 0). Selection is indicated not by a floating pill, but by standard active-state button fills and, crucially, the dynamic inclusion of an internal checkmark icon that displaces the label text.Material 3 also introduced complex shape-morphing physics upon press and selection, further distinguishing the individual button segments from a shared track. When a segment in a standard Material button group is pressed or selected, it dynamically alters its padding, widening itself and compressing adjacent buttons. In the specific "Connected" variant, the pressed button changes shape without affecting adjacent buttons, relying on standard 2dp padding to provide visual consistency.Required DOM AnatomyRegardless of the visual model chosen, the systematic anatomy relies on the following structural nodes:Container (Group): The bounding <div role="group"> or <ul> managing the flex layout, accessibility grouping, and standard width constraints.Segment (Item): The atomic <button> accepting text, icons, or a combination thereof.Indicator: The visual representation of the selected state. In track models, this is a dedicated DOM node that absolutely positions itself over the active segment. In connected models, this is the individual background CSS of the segment itself.Divider (Track Model Only): The vertical separator. Its visibility logic is highly state-dependent, hiding itself if the adjacent segment is selected, hovered, or currently managing keyboard focus.APIThe application programming interface for the Segmented Control must gracefully handle the fundamental behavioral fork: single-select (exclusive) versus multi-select. Design systems manage this dichotomy through differing architectural strategies, primarily relying on either distinct component exports or sophisticated prop interfaces. The practice establishes a unified component with a strict prop signature to handle both modalities.Single vs. Multi-Select SignaturesSystems such as MUI (v5) utilize a singular ToggleButtonGroup component and expose an exclusive boolean prop. When exclusive={true}, the component enforces radio-button-like mutual exclusivity. The value prop expects a scalar primitive (e.g., a string), and the onChange callback yields that single scalar value. When exclusive={false} (the default for multi-select), the component adopts checkbox-set mechanics. The value prop expects an array of primitives, and the onChange callback yields the updated array.Adobe's React Spectrum (v3) manages this similarly but utilizes a more explicit selectionMode prop, accepting the string literals "single" or "multiple", alongside selectedKeys and defaultSelectedKeys for controlled and uncontrolled states. Base Web (v12) configures this via a mode prop, accepting "radio" or "checkbox".The opinionated default favors the explicit string literal approach (selectionMode="single" | "multiple") over a boolean exclusive toggle. A boolean prop often obscures the return type signature for TypeScript implementations, whereas a literal union string allows for tighter conditional type inference, ensuring that the onChange handler is strictly typed to return either string or string based on the mode.The Zero-Selection TrapThe second major API consideration is the prevention of empty states. In many product scenarios, a view must have at least one active filter. Allowing a state where no segments are selected would result in a broken or empty user interface. The API must support enforcing a constant selection.In MUI, exclusive selection does not inherently force a button to remain active; a user can click an already active button to deselect it, yielding a null value. To prevent this, MUI requires the consumer to implement a manual guard clause within the onChange handler to reject null payloads. React Spectrum, recognizing this boilerplate, introduces a disallowEmptySelection boolean explicitly for this constraint.An opinionated default practice must provide disallowEmptySelection as a first-class prop. When disallowEmptySelection={true}, the internal click handler ignores events on the currently active segment, guaranteeing that the onChange callback never fires with a null or empty array value.Standard Prop InterfaceProp NameTypeDefaultDescriptionselectionMode"single" | "multiple""single"Dictates the fundamental semantic behavior, ARIA roles, and the data type of the value signature.valuestring | stringundefinedThe controlled value of the group. Failure to synchronize these states leads to accessibility violations and disjointed visual feedback.Macro-Level Group StatesAt the macro-level, the entire group can be marked as disabled. When a group-level disable is invoked, the component cascades the disabled property to all internal segments, immediately removing the group from the document's sequential focus navigation. Modern systems such as Base Web and MUI explicitly support both group-level and segment-level disabled states simultaneously. This allows application logic to disable the entire component during data fetching, while also allowing specific subset segments to be disabled based on granular user permissions, all while leaving the overarching control functional.Micro-Level Segment StatesAt the micro-level, the individual segments inherit the standard interactive lifecycle of a button, but with specific layout caveats introduced by the adjoined nature of the control:Rest: The segment is inactive. Contrast requirements dictate a strict 4.5:1 ratio for text against the track background, and a 3:1 ratio for borders against the page surface.Hover: The segment receives pointer intersection. The component often introduces a subtle background fill or modifies the opacity of the text. In the Material 3 Connected model, hover states must not bleed into adjacent buttons.Focus-Visible: The segment receives keyboard focus. A high-contrast focus ring must encompass the segment. If the track-and-pill visual model is used, the focus ring must align with the inner dimensions of the track rather than bleeding outside the container. If the Connected Button model is used, focus rings trigger "z-index wars" (detailed in the Implementation section), where the focused segment must dynamically elevate its z-index to prevent adjacent collapsed borders from clipping the focus outline.Pressed/Active: The pointer is depressed. Apple HIG and Material 3 both utilize shape morphing and opacity changes during the pressed state.Selected: The segment represents the active programmatic value. The indicator pill slides into position, or the button background shifts to its active color token. If the system utilizes Material 3 guidelines, an internal checkmark icon dynamically animates into the layout, requiring a complex layout shift to accommodate the new icon width without breaking the equal-width constraints.VariantsThe practice establishes variants based on density, content configuration, and sizing strategy. These variants dictate how the control responds to unpredictable content lengths and viewport constraints.Content Configuration VariantsThe primary content variants are Text-Only, Icon-Only, and Text-with-Icon.
When utilizing Icon-Only variants, GitHub Primer strictly mandates the use of an aria-label or visually hidden text, along with a mandatory tooltip, to ensure the abstract iconography remains comprehensible to both sighted and non-sighted users. Systems must forbid the mixing of content types within a single group; a Segmented Control must not feature one segment as Text-Only and another as Icon-Only, as this violates standard Gestalt principles of similarity and creates an imbalanced visual hierarchy. Every segment in the group must conform to the same content variant.Sizing Strategy VariantsThe sizing strategy introduces a fundamental conflict between equal-width distribution and content-width distribution.
The standard iOS specification dictates that all segments within a control should be of exactly equal width, determined by calculating the widest localized string in the entire set and applying that width to all siblings. This ensures absolute visual equilibrium and highly predictable spatial targeting.However, for responsive web applications, fluid layouts often necessitate a content-width approach, where each segment's width is dictated purely by its internal text plus padding. A strictly equal-width control on the web can lead to massive whitespace in shorter segments when placed next to a particularly long segment. The opinionated default is to enforce equal-width distribution (flex: 1 on all children) by default to maintain symmetry, only falling back to content-width (flex: auto) when explicitly overridden via a prop (e.g., isIntrinsic={true}) or when horizontal viewport constraints trigger responsive compression.AccessibilityThe accessibility architecture of the Segmented Control is historically fragmented. Because the HTML specification lacks a native <segmentedcontrol> element, engineering teams are forced to compose the pattern using disparate ARIA roles. This has led to four dominant, heavily contested implementation models across major enterprise design systems. Defining an opinionated practice requires resolving this "ARIA fork" based on the component's ultimate semantic purpose within the layout.Model 1: The Radio Group Paradigm (role="radiogroup")Material 3 and older iterations of web components map the single-select Segmented Control directly to a radiogroup containing radio elements. In this model, standard radio keyboard accessibility rules are strictly enforced. The group acts as a single tab stop in the DOM (utilizing a roving tabindex). When the user tabs into the group, focus lands exclusively on the currently selected segment. The user then navigates between options using the Left and Right arrow keys, and selection automatically follows focus (unless the system is explicitly configured for manual activation via Space/Enter).The critical flaw in the Radio Group paradigm is detailed extensively by GitHub Primer. Primer strictly prohibits treating a Segmented Control as a radiogroup. In classic web architecture, standard radio buttons imply delayed execution—a user changes the radio selection, and the interface remains static until a separate "Save" or "Submit" button is activated. A Segmented Control, by definition, triggers immediate, highly visible view mutations. When a screen reader announces "Radio", the user intuitively expects a deferred action; immediate DOM mutations can therefore cause severe cognitive dissonance, interrupting the screen reader and causing structural disorientation.Model 2: The Independent Toggle Paradigm (role="group" + aria-pressed)To circumvent the semantic implications of the Radio Group, MUI constructs the ToggleButtonGroup using a generic role="group" container. Inside this container, each segment is exposed as a standard button equipped with the aria-pressed attribute, signaling state without relying on form inputs.This approach completely alters the keyboard interaction model. Roving tabindex is abandoned. The user must press the Tab key to iterate sequentially through every single button in the group. Arrow keys do nothing. The Space or Enter key manually toggles the aria-pressed state. This model works flawlessly for multi-select arrays (where aria-pressed acts identically to a checkbox), but it introduces heavy keyboard friction for single-select arrays, forcing power users to tab exhaustively through the entire interface just to bypass the component.Model 3: The List & Current Paradigm (<ul> + aria-current)GitHub Primer resolves the cognitive dissonance of the Radio Group by completely abandoning form semantics. Primer constructs the Segmented Control as an unordered list (<ul role="list">), where each segment is a standard button or link wrapped in an <li>. The selected state is not conveyed via form values (aria-checked), nor by toggle states (aria-pressed), but rather through the aria-current="true" or aria-current="page" attribute.This model treats the Segmented Control strictly as a collection of view-switching commands. Keyboard navigation relies on the Tab key (similar to MUI), but the screen reader announcement cleanly communicates that the user is interacting with a list of immediate actions. Primer explicitly states that if the control routes to a new URL, the segments must use role="link"; if it updates the DOM in place, they must use role="button".Model 4: The Tablist Paradigm (role="tablist")IBM Carbon maps their ContentSwitcher to role="tablist" with role="tab" children. This maps beautifully to the roving tabindex and arrow-key navigation model, providing exceptional keyboard ergonomics. However, strictly speaking, W3C ARIA authoring practices dictate that a tab must logically control an associated tabpanel. Content Switchers generally filter data within an existing layout rather than displaying a dedicated panel, leading to semantic technicalities. IBM Carbon currently maintains active issues acknowledging this ARIA mismatch, debating whether to migrate to a different standard.The Opinionated ResolutionAn expert design system cannot adopt a one-size-fits-all ARIA model for the Segmented Control due to its dual nature as both a data input and a view switcher. Furthermore, ensuring that the touch area for segments responds correctly to minimum accessibility sizes requires a strict 24x24 pixel inner bounding box minimum, as dictated by Primer's mobility guidelines.ContentWriting effective content for a Segmented Control requires militant brevity. Because horizontal space is the primary physical constraint of this component, character counts directly impact structural stability and responsiveness.The practice enforces a strict segment-count limit of two to five items for text-labeled controls. Exceeding five items guarantees layout failure on narrow viewports, causing text truncation, unreadable scaling, or unacceptable multi-line wrap behaviors. If a user interface demands six or more options, the Segmented Control is the incorrect pattern, and a Select menu must be utilized to preserve the layout. An exception to this rule is granted exclusively for Icon-Only Segmented Controls, which may expand to a maximum of six choices due to the fixed, minimal, and predictable dimensions of standard iconography.Content formatting dictates that segment labels must be nouns or short noun phrases (e.g., "Daily", "Weekly", "Monthly"). Verbs and action phrases (e.g., "View Daily", "View Weekly") introduce redundant syllables and inflate the character count unnecessarily. Multiline labels are strictly prohibited.Every Segmented Control must also possess an overarching group label to provide cognitive context to assistive technologies. If the label is not visible in the UI, an aria-label or aria-labelledby attribute must be placed on the container (e.g., aria-label="File View"). Without this, a screen reader user simply hears a list of disconnected terms without understanding the categorical grouping.MotionThe defining micro-interaction of the modern Segmented Control (specifically in the Track-and-Pill model) is the "Sliding Pill"—the visual active-state indicator that physically glides across the track from the previously selected segment to the newly selected segment.This motion provides critical spatial context, confirming to the user that their selection has been registered and establishing the relative location of the options. It grounds the component in physical reality. However, calculating the sliding pill relies on dynamic measurements of the DOM. Because segment widths can vary based on content and localization, the indicator element cannot rely on hard-coded CSS. It must receive explicit transform: translateX(x) and width values derived dynamically via JavaScript bounding boxes.The Reduced Motion ContractWhile the sliding pill is an elegant affordance, large, fast swiping motions on the horizontal axis are known medical triggers for users with vestibular (inner ear) disorders, potentially causing dizziness, headaches, or nausea. Consequently, an expert implementation must strictly bind the sliding animation to the user's operating system preferences via the prefers-reduced-motion media query.Mantine provides an optimal blueprint for this execution. When the operating system flag for reduced motion is detected via @media (prefers-reduced-motion: reduce), the system must immediately strip the transition-duration property from the indicator (e.g., transition: none!important), forcing the pill to snap instantly to the new location without any intermediate tweening. Alternatively, a subtle cross-fade opacity transition can replace the spatial slide, preserving the visual feedback of state change without the inner-ear triggering movement. This is a non-negotiable accessibility contract for modern web motion.I18nInternationalization (i18n) exposes the fragility of the Segmented Control. A component carefully balanced for equal-width English strings will structurally collapse when translated into languages with extreme character inflation, such as German or Russian.The practice dictates that Segmented Controls must never utilize text wrapping; a segment that wraps to a second line breaks the component's strict vertical alignment, creates uneven touch targets, and destroys the visual metaphor of the unified track. To manage localization expansion, the component must be capable of transitioning from an equal-width distribution (flex-basis: 0; flex-grow: 1) to an intrinsic content-width distribution, allowing longer words to consume proportional space based on their localized string length. If the total width exceeds the container, the component must handle overflow gracefully. Standard scrollable overflow within a Segmented Control is generally discouraged on desktop interfaces but remains standard practice on mobile viewports.For RTL (Right-to-Left) languages such as Arabic and Hebrew, the logical ordering of the segments must automatically reverse. This is natively handled by standard CSS Flexbox configurations (flex-direction: row naturally reverses in RTL document contexts), but the mathematical calculations for the sliding pill's translateX property must be dynamically inverted based on the document's dir attribute. If the dir="rtl", the offsetLeft calculations must be measured from the right edge of the container to ensure the motion occurs in the correct direction. React Spectrum specifically documents automatic RTL flipping for action groups and their internal padding calculations.ImplementationThe engineering architecture of the Segmented Control is riddled with implementation traps, specifically regarding styling isolation, DOM structure, border management, and lifecycle hooks.The Shared Border TrapThe Material 3 Connected Button Group implementation exposes a notoriously difficult CSS challenge: shared borders. When atomic buttons with standard 1px borders are placed side-by-side in a flex row, their touching borders double up at the intersections, resulting in a visually heavy 2px interior border. Resolving this requires complex CSS adjacent sibling combinators (+ or ~). The standard fix applies a negative left margin to overlapping siblings (margin-left: -1px) to collapse the shared borders and simultaneously resets the inner border radii to zero to create the connected visual block.Focus Ring Z-Index WarsFurthermore, these collapsed borders complicate focus-visible rings. If a middle button in a connected group receives keyboard focus, its outline or box-shadow focus ring must elevate above its siblings. Because of the negative margin overlap, the adjacent buttons will visually clip the focus ring of the active button. Resolving this requires dynamically adjusting the z-index of the focused or hovered segment during the interaction event, physically elevating it within the stacking context to ensure the focus ring renders cleanly over the adjacent collapsed borders.Pill Measurement LifecycleThe Track-and-Pill model requires an entirely different DOM structure. The indicator pill cannot be implemented as a ::before or ::after pseudo-element of the active button, because transitioning a pseudo-element across different DOM nodes is impossible in CSS. The pill must exist as a dedicated <div aria-hidden="true"> sibling to the button list, floating entirely independent of the button hierarchy. It requires a React useEffect (or framework equivalent) to read the offsetWidth and offsetLeft of the active segment and apply those measurements directly as inline styles to the pill.To prevent memory leaks and ensure responsiveness, this implementation must hook into a ResizeObserver rather than relying solely on global window resize events. A ResizeObserver ensures the pill accurately tracks its parent if the container is resized indirectly by a CSS grid, a flex layout shift, or a localized text change, ensuring the pill never misaligns from its target segment.CompositionSegmented Controls are rarely utilized as standalone, top-level architectural elements; they act as compositional sub-units within larger layout constructs.They are heavily deployed within Action Bars and Toolbars, serving as high-visibility mode-switchers. Adobe React Spectrum relies heavily on the ActionGroup component within its ActionBar implementation, providing a semantic grouping for mass data manipulation and localized filtering. In data-heavy enterprise applications, Segmented Controls are frequently embedded within Data Table headers or Card headers, allowing users to rapidly toggle the view format (e.g., Grid vs. List) or filter the immediate dataset (e.g., 30 Days, 60 Days, 90 Days).Because they are compositional by nature, they must not fight their parent container for spatial dominance. The fullWidth boolean API allows the Segmented Control to act responsively, filling the remaining space of a CSS Grid area or Flex container. However, when placed in exceptionally wide viewports, an unrestrained fullWidth Segmented Control becomes unusable due to Fitts's Law. The physical distance between the visual label and the far edge of the segment becomes so vast that cognitive association fails. Therefore, parent compositions must enforce strict max-width constraints on the control, keeping the segments physically proximate and visually cohesive regardless of the monitor size.Notesnotes.unverified: While theoretical ARIA literature strongly advises against treating Segmented Controls as tablist ARIA roles (as the segments do not directly control standard tabpanel elements), practical field testing indicates that older screen reader technologies often process the tablist role with far more accurate spatial and state announcements than standard role="group" or role="radiogroup" implementations. IBM Carbon's continued adherence to the Tablist pattern, despite explicitly acknowledging the ARIA specification violation in their own GitHub issues , implies a pragmatic concession to legacy assistive tech. Engineering teams adopting strict role="group" models should conduct independent empirical validation with JAWS and VoiceOver to ensure the cognitive mapping remains intact for visually impaired users.
