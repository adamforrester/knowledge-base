# Empty State — field-truth study (Claude pass)

## 1. Framing
An Empty State is the **rendered content of a data container when there's nothing to show** — and it turns absence into guidance instead of a dead end. It is the fifth and final Feedback brief and **closes the Feedback category**. It inherits **Button** (the CTA — see button), **Icon**/illustration (the visual — see icon), **Link** (secondary/inline links — see link), and **lightly the live-region pattern Toast established** (only for the *dynamic no-results announcement*, §6). It is **composed by the Data components** — the empty state *is* the zero-state of a Table, List, Tree, search-results grid, or card.

Two framings define it:
- **The genre taxonomy (the headline) — the *why* drives the content + action.** The reason a container is empty determines everything: **first-use/onboarding** (nothing ever created — the most valuable, a feature's first impression), **no-results** (a search/filter returned nothing), **user-cleared/all-done** (inbox-zero, task complete), **error/failure** (couldn't load — the empty-vs-error boundary), **no-permission** (no access). Each maps to a *different* heading, body, and action.
- **The terminal-states boundary** — a data container has a lifecycle: **loading → {content | empty | error | no-permission}**. Empty State is one terminal state; an **empty result is *expected*** while an **error is a *failure***.

What it *isn't*: a **Loading** state (→ Skeleton/Progress — the *in-between*, not a terminal — see progress, skeleton when briefed), or an **Error** that should be a Banner/Inline message (a failed load is a failure, not expected absence — see banner, inline-message). Why systems disagree: the genre taxonomy, the illustration question, and the empty-vs-error treatment.

## 2. Anatomy and parts
- **illustration / icon** — the visual (illustrated / icon / none); **decorative** (`aria-hidden` — the meaning is in the text), §6.
- **heading** — a short, positive/neutral title (the essence — "No orders yet", not "Empty").
- **body / description** — explains *why* it's empty + *what to do*.
- **primary action** (inherits Button) — the genre-mapped CTA (create / retry / clear filters), §5.
- **secondary action** — an optional lighter affordance (learn more / a Link).
- **container** — the host region (full-page / in-card / in-row / in-section), §4.

## 3. Properties / API
- **parts/slots** — `illustration`/`icon`, `heading`/`title`, `description`/`children`, `action`/`primaryAction` + `secondaryAction` (composable parts — Chakra's `EmptyState.Root`/`Indicator`/`Title`/`Description`/`Content` — or a monolithic prop set — Polaris `EmptyState` with `heading`/`image`/`action`).
- **`variant` / genre** — informational, or a genre marker driving the default visual/tone (often left to content rather than a prop).
- **`image` / `imageContained`** — the illustration source; Ant's `image` (default vs `Empty.PRESENTED_IMAGE_SIMPLE`).
- **size** — `compact` (in-row/in-card) vs `full` (full-page).
- **API philosophy** — composable parts (slot the illustration/heading/body/actions) is the modern default; a monolithic prop set for the common case.

## 4. States and variants
- **By genre (the primary axis):** **first-use** (create CTA, encouraging) / **no-results** (clear-filters CTA, helpful) / **all-done** (often no CTA, reassuring) / **error** (retry CTA, no-blame) / **no-permission** (request-access CTA).
- **By visual:** illustrated / icon / text-only.
- **By placement/size:** full-page / in-card / in-table-row / in-section; compact vs full.

## 5. Usage guidance
- **Use an Empty State** whenever a data container could be empty — **never leave a blank container** (absence without guidance is a dead end). Match the **genre** to the situation and the **action** to the genre.
- **The genre → action mapping (the core rule):**
  - **First-use** → a **primary create CTA** ("Create your first project") + encouraging copy; treat it as **onboarding** — it's the user's first impression of the feature.
  - **No-results** → **clear-filters / adjust-search / broaden** ("No results for 'X' — clear filters") — help them recover, don't dead-end.
  - **All-done / user-cleared** → often **no action** (or a gentle next step) — reassure ("You're all caught up").
  - **Error** → **retry** + a no-blame explanation (or escalate to a Banner/Inline error if it's a hard failure).
  - **No-permission** → **request-access / contact-admin**.
- **Don't use an Empty State for:** the **loading** in-between (→ Skeleton/Progress — see progress), or a **failure that warrants a persistent error surface** (→ Banner/Inline — see banner; the error-flavored empty state is for *in-container* "couldn't load this list", not a page-level system error).
- **Decision frameworks:**
  - **Empty vs loading vs error:** expected-absence-with-guidance (Empty) vs in-progress (Skeleton/Progress) vs failure (Banner/Inline or an error-flavored empty). The container's state machine.
  - **Action discipline:** one primary CTA mapped to the genre; a secondary action only if it genuinely helps (don't pile on).

## 6. Accessibility
- **The empty-state content is *not* a live region** — it's the rendered content of a region, read in document order; it doesn't need `role=alert`/`status` just for existing (the Banner static-vs-dynamic lesson applies — static content isn't a live region).
- **The dynamic no-results announcement (the one live-region touch):** when a search/filter *empties the results in response to a user action*, the **result count** ("No results found", "0 results") should be announced via a **polite status live region** (reusing Toast's pattern lightly) — and/or the results region's accessible status updates — so a screen-reader user knows the query returned nothing. Resolve **focus**: typically **keep focus in the search field** (the user is iterating the query) rather than yanking it to the empty state; the empty state is then encountered/announced via the status update, not a focus move.
- **The heading in the outline** — the empty-state heading takes an appropriate `<h*>` level for its container's position (don't break the document outline — the heading-level-mapping lesson from Banner/Inline Message).
- **The illustration is decorative** — `aria-hidden="true"` (an empty inbox illustration adds no information AT needs); the heading + body carry the meaning.
- **The action is in the tab order** — a normal Button (no special wiring).
- **Color-not-sole-carrier** — the genre isn't conveyed by colour; the heading/body text is the signal.
- **WCAG SCs:** 4.1.3 (Status Messages — the dynamic no-results announcement), 1.3.1 (heading/structure), 1.1.1 (decorative illustration `aria-hidden`), 2.4.6 (headings/labels), plus the action Button's SCs. (A static empty state mostly just needs correct semantic structure.)

## 7. Content guidelines
- **Heading — positive/neutral, names the situation** — "No orders yet" / "You're all caught up" / "No results for 'invoices'", never "Empty" / "Nothing here" / "Error" (for the no-data case). Short.
- **Body — explain *why* + *what to do*** — "Orders will appear here once customers buy. Add your first product to get started." Concise.
- **Action label — a clear verb mapped to the genre** ("Create project", "Clear filters", "Try again", "Request access").
- **Tone by genre** — **encouraging** (first-use: an opportunity, not a void), **helpful** (no-results: recovery), **reassuring** (all-done: positive), **no-blame** (error: "We couldn't load this — try again", not "You did something wrong").
- **First-use is the highest-value copy** — it's onboarding; invest in it (the difference between a feature that gets adopted and one that looks broken on first open).

## 8. Motion and transition
Minimal. The empty state appears when content is absent (or when a query empties the results) — a quiet fade-in is enough; avoid drawing attention to the *absence* with heavy motion. If transitioning from a loading skeleton to an empty state, a gentle crossfade. Under `prefers-reduced-motion: reduce`, snap. No decorative animation on the illustration. (See 18-motion-foundations, 19-motion-implementation-web.)

## 9. Internationalization
- **RTL** — the layout mirrors (illustration/text alignment, the action row) via logical properties; the illustration should be **direction-neutral** or mirror cleanly (avoid illustrations with embedded LTR text or strong directional metaphors).
- **Text expansion** — the heading and body expand (+30–50%); they must **wrap** (multi-line is fine — it's not a constrained slot) without truncation; the layout (often centered) accommodates longer copy.
- **Localised** heading/body/action; the no-results query echo ("No results for 'X'") handles the user's term. (See 13-internationalization-and-rtl.)

## 10. Naming
The field splits on framing: **`EmptyState`** (Polaris, Atlassian, Chakra — names the *state*), **`Blankslate`** (Primer — the blank-canvas metaphor), **`IllustratedMessage`** (Spectrum — names the *form*, and unifies empty/error/no-results under one component), **`Empty`** (Ant — the terse zero-state). **The practice default: `EmptyState`** (names the state clearly, the most common name) with **composable parts** (illustration/heading/description/actions) and a **size** (compact/full); the genre is driven by content (heading/body/action) rather than a heavy variant enum, with the dynamic no-results announcement handled by the data host. Note the Spectrum insight — empty/error/no-results share a *form* (illustration + message + action), so a system may implement one `IllustratedMessage`-style primitive and vary the content; the practice keeps `EmptyState` as the name but acknowledges the shared shape with the error/no-results cases. Aliases to map: `Blankslate`, `IllustratedMessage`, `Empty`, `ZeroState`, `NoResults`.

## 11. Implementation notes
- **The data-container terminal-state machine** — a Table/List/grid renders one of `loading` (Skeleton/Progress) → `content` | `empty` | `error` | `no-permission`; the Empty State is the `empty` (and optionally `error`/`no-permission`) branch. Build the host to switch cleanly between these (and don't show an empty state *during* loading — that flash is the loading-state's job).
- **The no-results status announcement** — when a query empties the results, push a "No results found" (or the count) string to a **polite status live region** (a pre-existing region — the Toast rule), and **leave focus in the search field**; the empty state itself is just rendered content.
- **The decorative illustration** — `aria-hidden="true"` + `alt=""` (or a CSS background); never an informative `alt` for a generic empty-box illustration.
- **Heading level** — interpolate the correct `<h*>` for the container's position (a polymorphic level prop), like Banner/Inline Message.
- **Composable parts** — slot illustration/heading/description/actions (Chakra-style) so teams can compose first-use vs no-results vs error from one primitive; a monolithic prop set for the common case.
- **Don't hand-roll** — Polaris `EmptyState` / Primer `Blankslate` / Spectrum `IllustratedMessage` / Chakra `EmptyState`; skin them; wire the no-results announcement at the data-host layer.

## 12. Related and alternative components
- **Stands on:** Button (the CTA — see button), Icon/illustration (the visual — see icon), Link (secondary/inline links — see link), and lightly the **live-region pattern** from Toast (the dynamic no-results announcement — see toast).
- **Composed by:** Table / List / Tree / search-results / card — the data hosts whose zero-state *is* the empty state (see table/list/tree when briefed).
- **Alternative to / boundary with:** **Skeleton / Progress** (the *loading* terminal state — the in-between, not expected-absence — see progress, skeleton when briefed), **Banner / Inline Message** (the *error/failure* terminal state — a failed load that warrants a persistent error surface vs an in-container error-flavored empty state — see banner, inline-message).

## 13. Field POV evolution
1. **The genre taxonomy matured** — practice moved from a single generic "empty" to distinct genres (first-use / no-results / all-done / error / no-permission) each with its own content + action.
2. **Empty-state-as-onboarding** — the first-use empty state is recognised as a feature's first impression and a high-value onboarding surface, not an afterthought.
3. **The unified illustrated-message form** (Spectrum) — empty/error/no-results share a shape (illustration + message + action), so systems implement one primitive and vary the content.
4. **The no-results announcement settled** — a polite status live region + focus-stays-in-search, over silent result-emptying.
5. **Illustration restraint** — a pullback from bespoke per-state illustrations (cost, inconsistency) toward lighter icons / restrained illustration, with the illustration always decorative.
