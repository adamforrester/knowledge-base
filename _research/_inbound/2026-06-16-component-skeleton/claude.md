# Skeleton — field-truth study (Claude pass)

## 1. Framing
A Skeleton is a **content-shaped placeholder for an initial load** — gray shapes that mimic the layout and dimensions of incoming content (text lines, avatar circles, image/card rectangles). It **closes the three-loading-patterns triad** Progress and Spinner anchored: **Progress** (a value contract / measurable / longer), **Spinner** (minimal inline indeterminate, no shape), **Skeleton** (an *initial* known-layout load, **reserving the layout**). It inherits the **anti-flash (delay/min-display) + `aria-busy` region pattern from Progress/Spinner** (see progress, spinner) and lightly the **"Loading" live-region announcement from Toast** (see toast). It is also the **loading terminal state** of the data-container machine Empty State described (`loading → {content | empty | error}` — Skeleton is the loading branch; see empty-state).

Two things define it:
- **The layout-reserving / CLS contract (the Spinner opposite).** A Spinner gives *no shape*; a Skeleton *matches the final shape and dimensions* so when real content arrives there's **no Cumulative Layout Shift** — and the eye, already primed with the structure, perceives a shorter wait. The corollary is the **mismatch trap**: a skeleton whose shape/size *doesn't* match the real content causes a **worse jump than no skeleton** — shape-match fidelity is the whole point.
- **The reduce-motion difference from Spinner (the sharp distinction).** A Skeleton's information is the **shape**, not the motion — so unlike a Spinner (whose motion is the *only* cue and must be *substituted*), `prefers-reduced-motion: reduce` can simply **drop the shimmer/pulse to a static skeleton** (the shape still says "loading").

What it *isn't*: a **Spinner** (no shape; minimal/inline/short — see spinner), a **Progress** bar (a value/measurable task — see progress), or an **Empty State** (the *empty* terminal state — Skeleton is the *loading* one; see empty-state). Why systems disagree: the variant model, auto-derive-vs-hand-built, and the shimmer-vs-pulse animation.

## 2. Anatomy and parts
- **placeholder shapes** — the primitives: a **text line** (a rounded bar; multiple lines with a **shorter last line** to mimic a paragraph), a **circle** (avatar), a **rectangle/rounded** (image/card/thumbnail/button). Sized to match the real element.
- **base + highlight** — the placeholder's base fill + the animated highlight (the shimmer's brighter sweep band); theme tokens (light/dark — §9).
- **the skeleton screen** — composed: a whole section/page of skeleton shapes mirroring the final layout (Polaris `SkeletonPage`).
- **the `aria-busy` region wiring** — *not* a visible part: the **loading region** (the container the content will fill) carries `aria-busy="true"` + a status announcement; the skeleton shapes themselves are `aria-hidden` (§6).

## 3. Properties / API
- **`variant`** — text / circular / rectangular / rounded (MUI); or per-component skeletons (`SkeletonText`/`SkeletonAvatar`/`SkeletonImage`/`SkeletonButton` — Primer/Chakra/Ant/Carbon/Polaris).
- **`width` / `height`** — match the real element (or **auto-derive from children** — MUI infers dimensions from the wrapped element).
- **`lines` / paragraph** — the text-skeleton line count (+ the shorter last line).
- **`animation`** — pulse / wave(shimmer) / none(false) (MUI's `animation`).
- **`isLoaded` / `loading`** — the swap toggle (Chakra `isLoaded` wraps children and shows the skeleton until loaded).
- **auto-derive vs hand-built** — infer-from-children (MUI) vs a hand-built **skeleton screen** (Polaris `SkeletonPage` mirroring the layout) — the trade is fidelity-with-effort vs convenience (§5/§11).

## 4. States and variants
- **States:** **loading** (the only state — the skeleton is shown; on resolution it **swaps** to the real content). (+ the inherited anti-flash *delayed* / *min-display-locked* windows.)
- **Variants:**
  - **shape:** text (lines) / circular (avatar) / rectangular / rounded (image/card).
  - **primitive vs per-component** (a generic `Skeleton` with a `variant` vs `SkeletonText`/`Avatar`/etc.).
  - **the skeleton screen** (a composed layout of skeletons).
  - **animation:** shimmer/wave / pulse / static.

## 5. Usage guidance
- **Use a Skeleton** for an **initial page/section load where the layout is known/predictable** — lists, cards, profile headers, feeds, data tables, an image loading into a known box. It reserves the layout (no CLS) and primes the user with the structure.
- **Don't use a Skeleton for:**
  - A **minimal inline / short / unknown-layout wait** → a **Spinner** (a button, a dropdown fetching — you can't shape-match a single control's brief wait; see spinner).
  - A **measurable/longer task** → a **Progress** bar (see progress).
  - The **empty result** → an **Empty State** (Skeleton is the *loading* state, not the *empty* one; see empty-state).
- **Decision frameworks:**
  - **Skeleton vs Spinner (the core call):** known, shape-matchable layout on initial load (Skeleton) vs minimal/inline/short/unknown (Spinner). Primer's rule: a 20-widget dashboard uses *one* skeleton layout, not 20 spinners.
  - **Shape-match fidelity (the load-bearing rule):** match the real content's dimensions closely so the swap is seamless; a **mismatched skeleton is worse than none** (a bigger jump). Don't over-engineer per-pixel skeletons, but don't ship generic boxes that reflow on swap.
  - **The anti-flash rules (inherited):** **delay before showing** (~200–500ms — a fast load shows nothing) + **minimum display time** (~500ms once shown). Don't flash a skeleton for one frame.
  - **Don't skeleton everything** — reserve it for genuine structured-content loads; not every async sliver needs a skeleton screen.

## 6. Accessibility
- **The skeleton is presentational scaffolding, NOT content** — it must be **`aria-hidden="true"`** so a screen reader doesn't read "blank/gray box, gray box, gray box" (a meaningless flood). The shapes are visual structure, not information.
- **The region carries the state** — the **loading region** (the container the content will fill) gets **`aria-busy="true"`** (so AT knows it's loading and can hold partial-update announcements) + a polite **status announcement** ("Loading"), flipping to **`aria-busy="false"`** + the real content (and a "Loaded"/content announcement) on resolution. This is the same `aria-busy`-region pattern Spinner/Progress use, reused here. (The `aria-busy`-region + status is the practice default; the skeleton itself is never the announced element.)
- **Load-complete swap + focus** — on resolution, the content replaces the skeleton in place; don't move focus to the newly-loaded content (it didn't ask for focus) — announce via the status/`aria-busy` flip. Preserve any focus the user had.
- **Reduce-motion** — `prefers-reduced-motion: reduce` **drops the shimmer/pulse to a static skeleton** (the shape is the cue; no substitution needed — the key difference from Spinner, whose motion *is* the cue). A static skeleton is fully acceptable/accessible.
- **Contrast** — the skeleton base/highlight is decorative (no text-contrast requirement on the empty shapes), but the shimmer shouldn't be so high-contrast/fast as to be a distracting flash (and it respects reduce-motion).
- **WCAG SCs:** 4.1.3 (Status Messages — the loading/loaded announcement on the region), 1.1.1 (the skeleton is decorative — `aria-hidden`), 2.3.3/2.2.2 (the shimmer animation + reduce-motion), 1.4.13 (n/a — it's not hover content). (A skeleton mostly needs the `aria-hidden` + the region's `aria-busy`/status — it's the lightest a11y of the loading family because there's nothing to read.)

## 7. Content guidelines
- **The skeleton has no content** — it's scaffolding; **don't fake content** (no lorem-ipsum text inside it, no placeholder labels an SR could read — the shapes are blank and `aria-hidden`).
- **Shape-match guidance** — the *shape* should read as the content type (text lines for text, a circle for an avatar) so the user anticipates what's coming; the last text line is shorter (mimics a paragraph's ragged end).
- **The loaded announcement** — the *region* announces "Loaded"/the content arriving (the skeleton itself says nothing); confirm the wait ended for AT.
- **Don't over-skeleton** — a screen of shimmering boxes for a tiny load is noise; reserve it for genuine structured loads.

## 8. Motion and transition
- **Shimmer (wave) vs pulse vs static:** **shimmer** = a gradient highlight band sweeping across the shapes (the most common; implies "loading" with directional motion); **pulse** = the whole skeleton fading opacity in/out (gentler, cheaper); **static** = no animation (the shape alone). The practice default: a subtle shimmer or pulse, with **static as the reduce-motion fallback**.
- **The reduce-motion DROP (the Spinner difference)** — under `prefers-reduced-motion: reduce`, **simply stop the animation** → a static skeleton. The shape still communicates loading, so (unlike Spinner) no substitute motion is needed. This is the cleanest reduce-motion case in the loading family.
- **The content-swap** — when real content arrives, a quick crossfade (or instant swap) replaces the skeleton; the shape-match means no layout jump. The anti-flash delay/min-display govern *whether/how long* the skeleton shows.

## 9. Internationalization
- **RTL** — the **shimmer sweep direction flips** (an LTR left-to-right sweep becomes right-to-left), matching the reading direction; the **shape layout mirrors** (a leading avatar circle + trailing text lines flips) via logical properties.
- **Text-line skeletons** — the line count/width can suggest the (expanded) translated text length; don't hardcode widths that mismatch the localized content (the mismatch trap in i18n form).
- **Dark mode** — the base/highlight tokens flip for the theme (a light shimmer on a dark base). (See 13-internationalization-and-rtl.)

## 10. Naming
**`Skeleton`** is the dominant name (MUI, Ant, Carbon, Polaris, Primer, Chakra, Fluent, Base Web) — for both the primitive and the per-component family (`SkeletonText`/`SkeletonAvatar`/`SkeletonImage`); **`Placeholder`** / **`Shimmer`** / **`Ghost`** are older/looser aliases. **The practice default: `Skeleton`** as the primitive (`variant: text | circular | rectangular`, `width`/`height`/`lines`/`animation`), with **per-component skeletons** (`Skeleton.Text`/`.Avatar`/`.Image`/`.Button`) for the common shapes and a composed **skeleton screen** for whole layouts; the auto-derive-from-children convenience layered on (MUI). Aliases to map: `Placeholder`, `Shimmer`, `Ghost`, `SkeletonText`, `SkeletonAvatar`, `SkeletonImage`, `LoadingPlaceholder`.

## 11. Implementation notes
- **The `aria-hidden` skeleton + `aria-busy` region + status** — mark the skeleton shapes `aria-hidden`; set `aria-busy="true"` on the loading region + a pre-existing polite status ("Loading"); on resolution flip `aria-busy="false"`, render the content in place, and (optionally) announce "Loaded" (§6).
- **Shape-match-the-final-layout (CLS)** — the skeleton's `width`/`height` (and the text-line count) must equal the real content's so the swap is shift-free; **auto-derive from children** (MUI: wrap the real element, infer its box) for fidelity-with-low-effort, or hand-build a **skeleton screen** (Polaris `SkeletonPage`) for a bespoke layout. Mismatch = a worse jump than no skeleton.
- **The anti-flash timers (inherited from Progress/Spinner)** — a `delay` before showing (~200–500ms; if the load resolves first, render nothing) + a `minDisplay` lock (~500ms once shown); manage the overlapping timers.
- **The shimmer/pulse + reduce-motion drop** — a CSS gradient sweep (`background-position` animation) or an opacity pulse; under `prefers-reduced-motion: reduce`, **disable the animation** (static) — no substitution (the Spinner difference, §8).
- **The `isLoaded` swap** — a `Skeleton`/`isLoaded` wrapper renders the skeleton until the data resolves, then the children (Chakra); keep the swap in place (same box) for CLS.
- **Don't hand-roll** — MUI/Ant/Carbon/Polaris/Primer/Chakra `Skeleton`; skin them (they get the shapes/animation/`aria-hidden` right) and wire the `aria-busy`/status at the region/host layer.

## 13. Field POV evolution
1. **Skeleton-over-spinner for initial loads** — the field shifted from a centered spinner on a blank page to shape-matched skeleton screens for initial structured loads (better perceived performance + no CLS); the spinner retreated to inline/micro-interactions.
2. **The shape-match/CLS discipline** — with Core Web Vitals (CLS) as a metric, the layout-reserving purpose hardened: a skeleton must match the final dimensions, and a mismatched skeleton is recognised as worse than none.
3. **The `aria-hidden` + `aria-busy` + status settling** — practice converged on the skeleton-as-decorative (`aria-hidden`) + the region carrying `aria-busy`/status, over screen readers reading the empty boxes.
4. **The reduce-motion-drop-to-static** — recognised as distinct from Spinner: the skeleton's shape is the cue, so the animation can be removed entirely under reduce-motion (no substitution).
5. **Auto-derived skeletons** — inferring the skeleton from the real component's dimensions (MUI) emerged to keep skeletons in sync with content and reduce hand-built drift.
