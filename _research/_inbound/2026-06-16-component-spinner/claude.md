# Spinner — field-truth study (Claude pass)

## 1. Framing
A Spinner is the **minimal, indeterminate, inline busy indicator** — a small rotating mark that says "working, duration unknown" with **no value contract**. It is the small-inline corner of the **three-loading-patterns triad** the Progress brief anchored: **Progress** (a value contract / longer-or-measurable / large area), **Spinner** (minimal inline, ~1–5s, no value), **Skeleton** (a content-shaped placeholder for an initial load). It **inherits the anti-flash (delay/min-display) + reduce-motion-substitution + indeterminate-busy contract from Progress** (Spinner is the no-value inline sibling of an indeterminate Progress indicator — see progress), and lightly the **"Loading" live-region announcement from Toast** (`role=status`, not focus — see toast). Its single most common use is **composition by Button's loading state** (see button).

The defining contract is **no value** — a Spinner never carries `aria-valuenow`; it's pure "busy". And the architectural question it forces is the **Spinner-vs-indeterminate-CircularProgress overlap**: visually identical, so some systems ship a separate `Spinner` primitive (Polaris/Primer/Fluent/Chakra/Ant) and others fold it into `CircularProgress` (MUI) / `ProgressCircle` (Spectrum). **The practice ships a separate `Spinner`** — constrained to inline/small/no-value — precisely so a button loader isn't mistaken for a `progressbar` that needs a value contract.

What it *isn't*: a **Progress bar** (a measurable/longer task with a value — see progress; faking a determinate value is the Progress anti-pattern), or a **Skeleton** (a content-shaped placeholder for an *initial* load, reserving layout to prevent CLS — see skeleton when briefed; a spinner over a blank area gives no shape). Why systems disagree: separate-primitive-vs-variant, the announcement model, and inline-vs-overlay placement.

## 2. Anatomy and parts
- **rotating indicator** — the animated SVG arc/ring (a `stroke-dasharray` head spinning around a track, or a simple rotate); **`currentColor`** so it inherits the surrounding text/brand colour (§3).
- **label / tip text** — an optional visible "Loading"/contextual string; a **visually-hidden** accessible name when there's no visible text (§6).
- **overlay / mask** — for the block/section case: a scrim over the loading content (Ant `Spin` wrapping children), centring the spinner + optionally blocking interaction (§5).
- **the live-region wiring** — a `role=status` announcement ("Loading"/"Loaded"), §6.

## 3. Properties / API
- **`size`** — xs/sm/md/lg (scales to context: button-sized → page-sized).
- **`color`** — `currentColor`/inherit by default (adopts the context); a brand/contextual override; the track-vs-head treatment, thickness.
- **`label` / `aria-label`** — the accessible name (default "Loading"); **never omit** (§6).
- **`delay`** — the anti-flash delay before showing (Ant's `delay` — §5/§11).
- **overlay/nested-content** — a `spinning` boolean wrapping children with a mask (Ant `Spin`), or a standalone inline spinner.
- (No `value`/`min`/`max` — the no-value contract; if you need a value, it's a Progress.)

## 4. States and variants
- **States:** **spinning** (the only state — it's either present or not; no hover/focus/pressed — a presentational indicator). (+ the inherited anti-flash *delayed* and *min-display-locked* windows, §5.)
- **Variants:**
  - **size** (xs→lg).
  - **placement:** **inline** (in a button / next to text / in a small control) vs **centered/overlay/block** (a section/page loader, optionally masking content).
  - **with-label / bare** (bare requires a visually-hidden name — §6).
  - **in-button** (the loading-button composition — §5).

## 5. Usage guidance
- **Use a Spinner** for a **minimal, indeterminate, ~1–5s wait** where you can't measure progress and a content placeholder doesn't fit: a button submitting, a dropdown/menu fetching a small dataset, a small area refreshing.
- **Don't use a Spinner for:**
  - A **measurable or longer (>~5s) task** → a **Progress** bar (a value sets expectations; a spinner with no end is anxiety-inducing for long waits — see progress).
  - An **initial page/section content load** where the layout is known → a **Skeleton** (it reserves shape + prevents CLS; a spinner over a blank area tells the user nothing about what's coming — see skeleton when briefed).
- **Decision frameworks:**
  - **The three-loading-patterns decision (from the Spinner side):** minimal-inline-indeterminate (Spinner) vs value/longer (Progress) vs content-placeholder-on-load (Skeleton).
  - **The button-loading use (the most common):** a Button morphs to show a Spinner in-place — **width-preserving** (reserve the label's width so the button doesn't reflow/jump), the button **disabled/`aria-disabled` during load** (prevent double-submit), the label either kept beside the spinner or replaced; announce the loading→done transition (inherits the Button loading contract — see button).
  - **The anti-flash rules (inherited from Progress):** **delay before showing** (~200–500ms — if the op resolves first, never render → no flash) + **minimum display time** (~500ms once shown). A spinner flashed for 50ms feels *worse* than nothing.
  - **Inline vs overlay:** inline for a control/small area; a centered overlay (with a mask) for a whole section/page — but **overlays that block interaction** should be reserved for genuinely blocking operations (don't trap the user behind a mask for a background refresh).

## 6. Accessibility
- **The announcement (the contested point):** a spinner has no value, so the options are: **`role="status"` + a visually-hidden "Loading" label** (announced once when it appears — the common, recommended practice for a minimal spinner), **`role="progressbar"` without `aria-valuenow`** (the indeterminate-progressbar route — heavier, more apt when it's really an indeterminate *progress* indicator), or **`aria-busy="true"` on the region** being loaded (when the spinner overlays a content area — the region announces busy/done). **The practice default: `role="status"` + a "Loading" name for the standalone/inline spinner; `aria-busy` on the region for the overlay case.**
- **NEVER an unlabeled spinner** — a bare animated SVG is **silent to AT** (a sighted user sees motion; a screen-reader user gets nothing). It must carry an accessible name ("Loading", or contextual "Loading results") — a visually-hidden label or `aria-label`.
- **Reduce-motion can't remove the only cue** — the rotation *is* the busy signal, so `prefers-reduced-motion: reduce` **substitutes** (a slow opacity pulse, or a static indicator + "Loading…"), never deletes — a frozen spinner reads as a crash. (Inherited from Progress's indeterminate case.)
- **Focus/announcement on load-start/complete** — for the in-button case, don't move focus to the spinner; announce the state change (loading → loaded) via the status region; on completion, restore the button and (if focus was on it) keep it. For an overlay, `aria-busy` flips to false and the content is announced/available.
- **Color-not-sole-carrier** — the spinner's *motion* (not colour) conveys busy; the label is the explicit signal; the spinner must meet non-text contrast (3:1) against its background to be perceivable (WCAG 1.4.11).
- **WCAG SCs:** 4.1.3 (Status Messages — the "Loading" announcement), 1.4.11 (non-text contrast — the indicator perceivable), 2.3.3/2.2.2 (motion — reduce-motion), 1.1.1 (the visually-hidden name / decorative if `aria-busy` carries it), plus the Button's SCs for the in-button case.

## 7. Content guidelines
- **The label is "Loading"** by default, or **contextual** when it helps ("Loading results", "Saving") — concise.
- **The loaded announcement** — for longer/region cases, announce completion ("Loaded", or the result) so an AT user knows the wait ended, not just that it started.
- **Don't over-spin** — a spinner on every micro-interaction is visual noise; reserve it for genuine waits (and respect the anti-flash delay so fast ops show nothing).
- **In-button** — the label, if kept, stays ("Saving…"); if replaced, the visually-hidden/`aria` name persists so the button still has an accessible name while loading.

## 8. Motion and transition
The rotation is the spinner — typically an SVG ring with an animated `stroke-dasharray`/`stroke-dashoffset` (the arc grows/shrinks as it spins, the non-repetitive feel) or a simple continuous `rotate` (~0.6–1s/turn, linear or eased). Keep it smooth (GPU `transform: rotate`, not layout). Appear/disappear is a quick fade (with the anti-flash delay/min-display governing *whether* it appears). Under `prefers-reduced-motion: reduce`, **substitute** the rotation with a slow opacity pulse / static indicator (§6) — never a frozen ring. (See progress §8, 18-motion-foundations, 19-motion-implementation-web.)

## 9. Internationalization
- **RTL** — the rotation is direction-neutral (a spin reads the same either way; don't reverse it), but the **label sits on the correct side** of an inline spinner via logical properties, and an overlay/tip mirrors with the content.
- **The label localizes** ("Loading" → the locale string); keep it short.
- **`currentColor`** is locale/theme-neutral — the spinner adopts whatever colour context it's in. (See 13-internationalization-and-rtl.)

## 10. Naming
The field splits on the separate-primitive question: **`Spinner`** (Polaris, Primer, Fluent, Chakra, Base Web) and **`Spin`** (Ant — with the nested-content mask) ship it separately; **`CircularProgress`** (MUI) / **`ProgressCircle`** (Spectrum) / Material's **circular progress (indeterminate)** fold it into the progress family; **`Loading`**/`InlineLoading` (Carbon). **The practice default: a separate `Spinner`** — constrained to inline/small/no-value — so a button loader isn't treated as a `progressbar` needing a value contract (the React-Aria/Progress lesson: a spinner and an indeterminate `CircularProgress` are visually one thing but semantically the no-value end; a distinct `Spinner` name keeps the contract clear). The determinate/value case is **`Progress`** (see progress). Aliases to map: `Loading`, `InlineLoading`, `CircularProgress`, `ProgressCircle`, `Spin`, `Loader`, `BusyIndicator`.

## 11. Implementation notes
- **The no-value indeterminate markup** — `role="status"` + a visually-hidden "Loading" (standalone/inline), or `aria-busy="true"` on the region (overlay); **never** an `aria-valuenow` (that's a Progress). Don't put `role=progressbar` on a minimal button spinner.
- **The anti-flash timers (inherited from Progress)** — a `delay` `setTimeout` (~200–500ms; cleared on early resolution → no render) + a `minDisplay` lock (~500ms once shown); manage the overlapping timers (cleanup to avoid leaks).
- **Reduce-motion substitution** — branch on `prefers-reduced-motion` to the opacity pulse / static indicator.
- **In-button width-preservation** — reserve the label's width (or swap the label for the spinner at the same width) so the button doesn't reflow when it enters loading; disable + the loading announcement (the Button loading contract — see button).
- **The overlay/mask + interaction blocking** — Ant `Spin`-style: a scrim over the wrapped content, the spinner centred; **block interaction only for genuinely blocking ops** (`pointer-events`/`inert` on the masked content), and ensure the masked content is `aria-busy` (not just visually dimmed).
- **`currentColor` theming** — the spinner's stroke/fill is `currentColor` so it adopts the context (a white spinner in a primary button, the text colour inline) and works in light/dark via the inherited token — no separate assets.
- **Don't hand-roll** — Polaris/Primer/Fluent/Chakra `Spinner`, Ant `Spin`, or the system's indeterminate `Progress`/`CircularProgress`; skin them (they get `role=status`/the name/reduce-motion right).

## 12. Related and alternative components
- **Stands on:** the **anti-flash (delay/min-display) + reduce-motion-substitution + indeterminate-busy contract** inherited from Progress (Spinner is the no-value inline sibling — see progress), and lightly the **live-region pattern** from Toast (the "Loading" announcement — see toast).
- **Composed by:** **Button** (the loading state — the most common use; width-preserving, disabled — see button), and any control/area with a brief indeterminate wait.
- **Alternative to / boundary with:** **Progress** (the value/longer/measurable sibling — same family; faking a value is the Progress anti-pattern — see progress), **Skeleton** (the content-shaped initial-load placeholder, prevents CLS — see skeleton when briefed; a spinner gives no shape).

## 13. Field POV evolution
1. **The spinner-vs-circular-progress consolidation** — some systems folded the spinner into `CircularProgress`/`ProgressCircle` (MUI/Spectrum/Material) recognising they're one visual; others keep a separate `Spinner` for the no-value contract clarity. The practice keeps them distinct in *name/contract* even if they share rendering.
2. **The never-unlabeled hardening** — a bare animated SVG (silent to AT) is now a clear failure; `role=status` + a "Loading" name (or `aria-busy` on the region) is table stakes.
3. **The anti-flash discipline** (delay-before-show + min-display) reached spinners from the perceived-performance work — a 50ms spinner flash feels worse than none.
4. **Reduce-motion substitution** aligned — the rotation is reduced (opacity pulse), not removed (a frozen spinner reads as a crash).
5. **`currentColor` theming** became the default — one spinner that adopts its context (light/dark/brand) over separate per-theme assets.
