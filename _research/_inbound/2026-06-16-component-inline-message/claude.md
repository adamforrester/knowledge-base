# Inline Message — field-truth study (Claude pass)

## 1. Framing
An Inline Message is the **smallest, most contextual feedback surface** — a short message attached to a single control or scoped to a form section, that communicates validation, status, or help *in place*. It is the third Feedback brief and the one that **interlocks Feedback with Form**: the **field-level** inline message *is* the helper/error slot Text Field already established (wired to the control via `aria-describedby`, with `aria-invalid` on the field), so **Inline Message is the primitive that the form controls compose** — Text Field, Textarea, Select, Checkbox/Radio group, etc. all render an Inline Message in their message slot. It inherits **Text Field** (the describedby/invalid field-association chain + the helper↔error slot — see text-field), the **live-region announcement pattern + the static-vs-dynamic refinement** from **Toast** and **Banner** (see toast, banner), **Icon** (severity), and **Link** (error-summary/inline links).

It closes the live-region trio: Toast *established* the pattern (always dynamic), Banner *refined* it (static-vs-dynamic), and Inline Message *applies* it to **field association** — where the message is both wired to its control (`describedby`) and announced on appearance (live region), which is exactly where the contested double-announcement problem lives.

What it *isn't*: a **Banner** (a standalone, dismissible, section/page message — the inline message is tied to a *state* and is *not* user-dismissible — see banner), a **Toast** (transient, floating — see toast), or a bare **helper text** with no semantic association (the inline message is the *associated*, possibly-validated version). Why systems disagree: the announcement model, validation timing, and whether an error summary is required.

## 2. Anatomy and parts
- **severity icon** (inherits Icon) — error/warning/success/info; decorative (`aria-hidden`); **color-not-sole-carrier** (§6).
- **message text** — a short, specific, actionable string (§7).
- **optional links** (inherit Link) — an inline link in the message, or (for the error summary) a link to the offending field.
- **field-association wiring** — *not* a visible part: the message has an `id`; the **field** points at it via `aria-describedby` and sets `aria-invalid` (the chain inherited from Text Field — §6).
- **no dismiss** — unlike a Banner, an inline message is tied to a state (it appears/clears with the validation/condition), not user-dismissed.

Scopes: **field-level** (below/adjacent to one control — the validation/helper text) / **section-level** (above a fieldset / group of fields — a lighter, fill-stripped message; the Banner-inline boundary) / **form-level error summary** (§5/§6).

## 3. Properties / API
- **`severity` / `validationState`** — error (dominant) / warning / success ("looks good") / info (helper). Maps to icon + colour + (when dynamic) live-region politeness. Fluent's `validationState`, Ant's `validateStatus`, Carbon's invalid/warn.
- **`message` / `children`** — the text (+ optional inline Link).
- **field-association** — `id` (so the field's `aria-describedby` can target it) + the field's `aria-invalid`; **usually wired by the field/FormControl, not the message itself** (the message is the rendered content; the control owns the association — §11).
- **the live-region role** — `role=alert`/`status` only when the message is **dynamically inserted** (static-at-load needs none — §6).
- **error-summary composition** — a separate `ErrorSummary` (focusable, links to fields) that aggregates the per-field messages (§6).

## 4. States and variants
- **States:** the message is present/absent as a function of the field/section state; the **helper↔error slot swap** (the same slot shows neutral helper text, then the error replaces it on invalid — or both coexist, inherited from Text Field).
- **Variants:**
  - **severity:** error / warning / success / info(helper).
  - **scope:** field-level / section-level / form-level error summary.
  - **with-icon** / text-only.
  - **static** (server-rendered at load) vs **dynamic** (client validation on blur/submit) — the live-region fork (§6).

## 5. Usage guidance
- **Use a field-level Inline Message** for validation/help **attached to a specific control** — "Enter a valid email", "Password must be 8+ characters", or neutral helper ("We'll never share this"). It's the form control's message slot.
- **Use a section-level Inline Message** for a state scoped to a **group of fields or a form section** — lighter than a page Banner, fill-stripped, sitting above the section.
- **Use an Error Summary** at the top of a **long/complex form** when it's submitted with errors (§6).
- **Don't use an Inline Message for:** a **standalone, dismissible, page/section announcement** (→ Banner — see banner), a **transient confirmation** (→ Toast — see toast), or feedback **not tied to a specific field/section** (→ Banner).
- **Decision frameworks:**
  - **Inline Message vs. Banner:** tied-to-a-field/section-state + not-dismissible (Inline Message) vs. standalone + dismissible + page/section (Banner). The section-level inline message and the inline Banner variant converge here — keep the field/section-state-tied case as Inline Message, the standalone announcement as Banner.
  - **Validation timing (the central UX call):** **reward-early, punish-late** — *don't* error a field the user hasn't finished or hasn't touched (on-change/eager validation that errors mid-typing is hostile); validate **on blur** (after the user leaves the field) and **on submit** (catch-all), then **re-validate on change** *once a field is already in error* (so the error clears as soon as they fix it). On-submit-only is the safe floor; on-blur + re-validate-on-change is the practice default.
  - **When an error summary:** long/complex/multi-section forms, or any form where errors may be off-screen — the summary gives a single focus target + jump links.

## 6. Accessibility
This brief sits where Feedback's live-region pattern meets Form's field-association chain, and the **announcement model is the crux** (and the contested point):

- **The three patterns and the practice default:**
  - **(1) `aria-describedby` on the field** → the message is announced **when the field receives focus** (good for help text + persistent errors a user tabs back to; doesn't announce on *appearance*).
  - **(2) a live region (`role=alert`/`status`) on the message** → announced **when it appears** (good for a dynamic error on blur/submit; doesn't re-announce on field focus).
  - **(3) both** → announces on appearance *and* on focus — but risks a **double announcement** (and clutter).
  - **The practice default:** use **`aria-describedby` for the association** (so the error is read whenever the field is focused — the durable wiring) **plus a live region *only* for the dynamic appearance** when the error is inserted after load, and **avoid wiring both in a way that double-announces** (e.g. don't make the describedby target *also* an assertive live region that fires on focus). Set **`aria-invalid="true"`** on the field while it's in error.
- **The static-vs-dynamic refinement (from Banner) applied to fields:** a **server-rendered error present at page load** is read in document order (and via describedby on focus) — **no live region**; a **client-validation error inserted on blur/submit** is a **live region** (`role=alert`/assertive for errors) so it announces. Same rule Banner established; here it governs the field message.
- **The error-summary pattern (GOV.UK canonical) + focus management:** on submit-with-errors, render a **focusable error summary** at the top (a labelled region, often `role=alert` or focused programmatically) listing each error as a **Link that moves focus to the offending field**; **move focus to the summary on submit** so a screen-reader/keyboard user is told what failed and can jump to each field. Each field still has its own describedby'd inline message.
- **Color-not-sole-carrier (1.4.1)** — severity is icon + text (a red border alone fails); error text is the explicit signal.
- **WCAG SCs:** **3.3.1 (Error Identification — the error is identified in text)**, **3.3.3 (Error Suggestion — say how to fix it)**, 4.1.3 (Status Messages — the dynamic live region), 1.4.1 (colour not alone), 3.3.2 (labels/instructions — helper), 1.3.1 (the association), plus 4.1.2 (`aria-invalid`/`describedby`).

## 7. Content guidelines
- **Specific + actionable — what's wrong AND how to fix it** — "Enter a valid email address" / "Password must be at least 8 characters", never "Invalid" / "Error" / "This field is required" (when something more specific is possible). State the requirement, not just the failure (WCAG 3.3.3).
- **No blame, plain language** — "Enter your date of birth", not "You failed to…"; calm and helpful, not alarming.
- **Positive validation** sparingly — "Username available" where confirmation genuinely helps; don't success-toast every valid field (noise).
- **Helper text** (neutral, pre-error) states the requirement up front so the user avoids the error ("Must include a number"); the error then echoes/refines it.
- **Severity tone** matches the state; keep it to one short line where possible.

## 8. Motion and transition
Minimal. The risk is **layout shift**: a field error appearing **pushes the form content down** and disorients the user mid-form. **Reserve space for the message slot** (or animate a short height/opacity reveal ~100–150ms) so the form doesn't jump as errors appear/clear — especially on a multi-field form validating on blur. Under `prefers-reduced-motion: reduce`, snap. No decorative motion. (See banner §8, 18-motion-foundations, 19-motion-implementation-web.)

## 9. Internationalization
- **RTL** — the severity icon flips to the trailing/leading edge via logical properties; the message aligns with the field's text direction.
- **Text expansion** — error messages expand (+30–50%) and are often longer than a label; they must **wrap** (multi-line is fine) without truncation, and the reserved space/layout must accommodate two lines.
- **Localised + the error-summary order** follows reading order; numbers/formats in messages localise. (See 13-internationalization-and-rtl.)

## 10. Naming
The field is fragmented and mostly *attribute*- or *slot*-named rather than a standalone component: **`InlineError`** (Polaris), **`HelpText`** (Spectrum — neutral+error in one), **`FormHelperText`** + `error` (MUI), **`ErrorMessage`**/`HelperMessage` (Atlassian), **`validationMessage`**/`validationState` (Fluent), **`help`**+`validateStatus` (Ant), **supporting text** (Material). **The practice default: `InlineMessage`** as the primitive (severity + icon + text + the field-association hooks), composed by the form controls as their **`HelperText`/`ErrorMessage`** slot, with a separate **`ErrorSummary`** for the form-level pattern, and the section-level case as a lighter `InlineMessage` (vs the standalone `Banner`). Aliases to map: `InlineError`, `HelpText`, `FormHelperText`, `ErrorMessage`, `FieldError`, `ValidationMessage`, `SupportingText`.

## 11. Implementation notes
- **The field owns the association; the Inline Message is the rendered content.** The form control (or its `FormControl`/`Field` wrapper) generates the message `id`, points the input's `aria-describedby` at it, and toggles `aria-invalid` — the Inline Message component renders the icon + text. (This is the Text Field describedby-chain wiring; Inline Message is the message, not the wirer — see text-field §implementation.) The same primitive renders neutral helper or error by severity.
- **The static-vs-dynamic live region** — wrap/assign `role=alert`/`status` *only* when the message is inserted dynamically (client validation); a server-rendered error at load needs none (it's in reading order + describedby). Avoid the double-announce (describedby target that's also a firing assertive region).
- **Validation timing + re-validation** — validate on blur + on submit; **re-validate on change once a field is already in error** so it clears live; never error untouched fields. (Reward-early, punish-late.)
- **The error summary + focus on submit** — on submit-with-errors, render the summary, **move focus to it**, and wire each summary Link to focus its field (`href="#field-id"` / programmatic focus). Keep per-field messages too (the summary is additive, not a replacement).
- **Reserve space for the message** to avoid layout shift (§8).
- **Don't hand-roll the wiring** — React Aria / the headless `Field`/`FormControl` primitives (React Aria, Radix, Base UI, Ark) generate the `id`/`describedby`/`invalid` correctly; the Inline Message slots into them. Skin them.

## 12. Related and alternative components
- **Composed by:** Text Field, Textarea, Select, Combobox, Checkbox/Radio group — every form control renders the field-level Inline Message as its helper/error slot (the primitive; see text-field, textarea, select, combobox, checkbox, radio).
- **Stands on:** Text Field (the `aria-describedby`/`aria-invalid` field-association chain + the helper↔error slot — see text-field), the live-region pattern + static-vs-dynamic from Toast/Banner (see toast, banner), Icon (severity — see icon), Link (error-summary/inline links — see link).

- **Aggregated by:** the **Error Summary** (form-level) and the **Form** composite (see form when briefed).

## 13. Field POV evolution
1. **The announcement model settled toward describedby + a scoped live region** — over the older blanket-`role=alert`-on-every-field-error (which double-announces) — describedby for the association, a live region only for dynamic appearance.
2. **The error-summary pattern spread** from GOV.UK into mainstream systems as the accessible answer to long-form validation (focusable summary + jump links + focus-on-submit).
3. **Reward-early / punish-late validation timing** hardened — on-blur + re-validate-on-change, never error untouched/mid-typing fields.
4. **The static-vs-dynamic clarification (from Banner)** reached field errors — server-rendered errors aren't live regions; client-inserted ones are.
5. **Inline Message recognised as the primitive fields compose** — rather than each control re-implementing its own error text, the message + association is one shared primitive.
