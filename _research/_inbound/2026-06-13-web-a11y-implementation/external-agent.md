Architectural Web Accessibility Implementation for Design Systems: Primitives, Patterns, and Annotation Contracts1. The Accessibility Primitives the Web Platform ExposesThe foundation of any accessible design system relies on the strict and intentional utilization of the web platform's accessibility primitives. For a design system architecture, the primary directive is to "build on the element, not above it". Native HTML semantics serve as the first line of defense, providing built-in keyboard event handling, focus management, and implicit state reporting to assistive technologies without requiring supplemental JavaScript. A native HTML <button> element, for instance, inherently understands how to process both Enter and Space keydown events, naturally participates in the document's sequential focus navigation order, and broadcasts its semantic identity to the platform's accessibility API. When design systems deviate from these native elements—often to bypass stubborn user-agent styling or to construct highly custom composite controls—they must manually recreate these underlying primitives, introducing substantial implementation risk and testing overhead.ARIA 1.3 Surface Area and ARIA-in-HTML RulesWhen native HTML is insufficient to describe complex interactive widgets, the Accessible Rich Internet Applications (WAI-ARIA) specification provides the necessary ontological bridge. ARIA 1.3, an active editor's draft targeted for expected completion in 2026, introduces critical extensions to the semantic surface area. The specification divides semantics into roles (identifying the component type), states (identifying transient conditions such as aria-expanded or aria-busy), and properties (identifying static relationships such as aria-controls).ARIA 1.3 introduces several new roles and attributes designed to modernize collaborative web applications. For example, role="suggestion", role="comment", and role="mark" allow systems to programmatically communicate text revisions, deletions, and insertions, mirro the functionality of native <del> and <ins> elements but with expanded application for rich text editors. Additionally, aria-description provides a mechanism to programmatically describe an element without requiring a visible text node to reference via aria-describedby. However, testing in mid-2026 reveals that engine support for aria-description remains heavily fragmented, functioning reliably only on NVDA and iOS VoiceOver.The implementation of ARIA is governed by strict authoring rules. Abstract roles (e.g., roletype, widget, structure) serve strictly as the ontological foundation for the specification and must never be used by authors; only concrete roles (e.g., button, dialog, combobox) are valid in HTML. Furthermore, the ARIA-in-HTML specification dictates strict role conflict resolution mechanisms, most notably the sixth presentational rule. This rule mandates that the use of presentational roles (role="none" or role="presentation") must not conflict with the implicit functionality of the element. If an element is rendered focusable (e.g., via tabindex="0") or is assigned a global ARIA attribute (such as aria-label), the browser's engine resolves the conflict by ignoring the presentational role entirely, thereby exposing the element to the accessibility tree. This fail-safe ensures developers cannot inadvertently create "ghost" elements that receive keyboard focus but report no semantic identity to screen readers.The Accessibility Tree and Engine ArchitectureBrowsers do not pass the Document Object Model (DOM) directly to screen readers. Instead, they compile an accessibility tree—a synthesized model derived from the DOM, computed CSS styles, and ARIA attributes. This tree exposes four primary properties for each node: the computed accessible name, the accessible description, the role, and the state.The architecture of this compilation process differs significantly across the four major browser engines, leading to divergence in how assistive technologies interpret the interface. Chromium (Blink) employs a highly complex, dual-tree architecture. When a DOM mutation occurs, Blink utilizes its internal accessibility tree to check whether the mutation affects semantics (such as name, role, or geometry). If it does, Blink marks the corresponding node as "dirty". This hidden internal tree is then processed and serialized into the public platform-neutral BrowserAccessibility tree, which is ultimately queried by screen readers like JAWS and NVDA.Conversely, WebKit (Safari) and Gecko (Firefox) manage their tree updates through different event-dispatching mechanisms. In complex single-page applications where rapid DOM mutations occur, these architectural differences can lead to race conditions. For example, Gecko may prioritize certain layout recalculations differently than Blink, meaning that a dynamically injected aria-label might be read instantly by JAWS on Chromium but face microsecond delays on VoiceOver with WebKit, potentially leading to clipped announcements.The accessible name computation itself is a frequent source of engine divergence. The computation algorithm follows a strict hierarchy: aria-labelledby overrides aria-label, which overrides native attributes (like alt or title), which overrides the element's inner text content. A common pitfall in design systems is applying aria-label to non-interactive content (like a <div> or <span>). Because non-interactive elements are typically pruned from the accessibility tree to optimize performance, adding an aria-label to a generic container without a corresponding widget role often results in the label being silently ignored by the engine.The Accessibility Object Model (AOM) and ElementInternalsThe Web Components specification, combined with the emerging Accessibility Object Model (AOM), fundamentally shifts how design systems encode accessibility primitives into custom elements. Historically, applying default ARIA semantics to a custom Web Component required attaching attributes directly to the host node or manually juggling string-based ID references across Shadow DOM boundaries, which broke encapsulation and created brittle architectures.The ElementInternals API, widely available across all baseline browsers since late 2023, resolves this by exposing the ARIA mixin. This interface allows components to define default semantics natively via reflected properties. For example, a custom element can utilize this.attachInternals().role = 'checkbox' to establish an implicit role without exposing a physical role attribute in the DOM. This guarantees that designers and engineers consuming the component cannot easily strip its default semantic identity.Furthermore, ElementInternals supports IDREF reflection properties such as ariaDescribedByElements, ariaLabelledByElements, and ariaOwnsElements. Rather than relying on string-based IDs—which are difficult to generate uniquely and dynamically—these properties accept arrays of actual DOM element references. This capability allows a custom input component to associate its description directly with a node inside its own Shadow DOM, bypassing the historical limitation that aria-describedby string references cannot cross shadow boundaries.Common Pitfalls in Primitive ImplementationDesign systems frequently encode anti-patterns when attempting to over-engineer accessibility.Redundant Roles: Applying role="button" to a <button> or role="list" to a <ul> forces the accessibility tree to perform redundant recalculations and can, in some engine edge cases, override native behavioral heuristics.Role Override on Semantic Elements: Applying a widget role to an element that already possesses strong native semantics (e.g., placing role="button" on a <summary> element inside <details>) destroys the native state reporting (like the expanded/collapsed state native to <details>) and forces the screen reader to treat the complex element as a simplistic button.ARIA Without Keyboard: Adding role="button" to a <div> provides the semantic identity but provides zero keyboard operability. Without explicitly binding keydown event listeners for the Enter and Space keys, and without adding tabindex="0", the ARIA primitive is entirely useless to a screen reader user navigating via keyboard.aria-hidden on Focusable Elements: A critical failure occurs when a component applies aria-hidden="true" to a container that includes focusable descendants. When a user tabs to a focusable child within an aria-hidden parent, the screen reader's virtual cursor is placed on an element that theoretically does not exist in the accessibility tree, causing severe engine conflicts, silence, or application crashes.2. Focus and Keyboard ManagementFocus management represents the mechanical core of an accessible web application. The design system must dictate how focus moves, how it is trapped, and how it is visually represented across all interactive components.The Native Focus Model and tabindexThe native focus model relies on the tabindex attribute. An element with tabindex="0" is inserted into the natural document tab order, determined by its physical placement in the DOM hierarchy. An element with tabindex="-1" is explicitly removed from the sequential tab order but retains its ability to receive programmatic focus via JavaScript's .focus() method.The use of positive tabindex values (e.g., tabindex="1", tabindex="5") is fundamentally broken in modern component-driven architectures. Positive tabindex values force the browser to prioritize those elements over the natural DOM flow, stripping control from the user agent and creating unpredictable, jumping navigation paths for keyboard users. A robust design system strictly forbids positive tabindex values through static analysis tooling.Focus Indication and the :focus-visible PrimitiveDesign systems must encode focus indication tokens that balance brand aesthetics with strict WCAG compliance. Historically, the :focus CSS pseudo-class applied a focus ring regardless of the input modality, leading to aesthetic complaints from designers when mouse users clicked a button and the ring lingered. This often led to the hostile anti-pattern of outline: none.The :focus-visible pseudo-class serves as the modern system primitive for this requirement. Unlike :focus, :focus-visible utilizes sophisticated browser heuristics to determine whether the user is interacting via a keyboard or a pointing device. If a user clicks a button, the browser suppresses the :focus-visible ring; if the user tabs to it, the ring appears. This distinction allows design systems to implement robust, high-contrast focus rings for keyboard users without compromising the visual design for mouse or touch users. Additionally, the :focus-within pseudo-class allows a parent container (like a complex search bar) to display a focus ring when any of its descendants receive focus, providing critical visual feedback for composite widgets.Inertia, Modality, and Focus TrappingWhen rendering modal dialogs or blocking overlays, the system must prevent users from interacting with the background content. The inert attribute provides a declarative, native mechanism for disabling user interaction across entire subtrees of the DOM. Before inert reached baseline availability in 2023, design systems relied on fragile polyfills that recursively applied aria-hidden="true" and tabindex="-1" to all background elements. The inert primitive natively removes elements from the tab order, prevents pointer events, and explicitly prunes the content from the accessibility tree, seamlessly trapping screen reader cursors and keyboard focus within the active overlay.The web platform currently offers two distinct primitives for overlay management, which dictate when a manual focus trap must be written:<dialog> and showModal(): When a native <dialog> is invoked via the showModal() API, the browser automatically renders the surrounding document inert, moves focus to the first focusable element inside the dialog, and traps subsequent Tab keystrokes within the dialog boundaries. It also inherently handles the Escape key to dismiss the overlay.The Popover API (popover): Introduced as a baseline feature in 2025, the Popover API is designed for non-modal interactions (such as tooltips, dropdown menus, and teaching UI). Unlike a modal dialog, popovers do not trap focus, do not automatically move focus into the popover content, and do not render the background inert.If a design system utilizes the Popover API to construct a complex, interactive mega-menu, the system must manually implement focus management logic to ensure keyboard users can navigate into the popover and return safely to the trigger element when it closes.Keyboard Interaction Patterns for Composite WidgetsThe ARIA Authoring Practices Guide (APG) defines specific, mandatory keyboard interaction models for composite widgets (grids, listboxes, tablists, trees, menus). Design systems must encode these patterns globally to provide predictable interactions. A fundamental rule is that a composite widget should represent a single tab stop in the document flow; once focus enters the widget, the user navigates internal items utilizing the arrow keys.There are two primary architectural approaches to managing this internal focus:Roving tabindex: The system dynamically updates the tabindex of the child elements. Only the currently active item possesses tabindex="0", while all others are set to -1. When the user presses an arrow key, the system intercepts the event, moves the tabindex="0" state to the adjacent item, and explicitly calls .focus() on it. This approach guarantees that physical DOM focus aligns with visual focus, ensuring that if a user tabs away from the widget and returns, focus lands precisely where they left off.aria-activedescendant: The system maintains physical DOM focus on the parent container (e.g., the role="combobox" input) while dynamically updating the aria-activedescendant attribute to match the ID of the visually highlighted child element.While aria-activedescendant avoids the performance overhead of moving physical focus, it is notoriously brittle across assistive technologies. Certain screen readers, particularly Narrator on Chromium-Edge and TalkBack on Android, frequently fail to announce virtual focus updates unless the parent and child roles strictly match a predefined, narrow list of valid APG relationships. Roving tabindex is universally supported by the underlying accessibility trees of Blink, Gecko, and WebKit, and is the recommended default mechanism for design system implementations.Landmark Navigation and Skip LinksDespite the widespread adoption of structural ARIA landmarks (<main>, <nav>, <header>), the implementation of "skip to main content" links remains a mandatory practice. Screen readers easily navigate by landmarks, but sighted keyboard users rely entirely on the Tab key. Without a skip link, a keyboard user must tab through every single global navigation link before reaching the page content.A design system should encode the skip link as the very first focusable element in the document. It must remain visually hidden (typically via CSS clip-path or absolute positioning off-screen) until it receives keyboard focus, at which point it must become visible and meet standard contrast requirements. The target of the skip link (the <main> element) must possess tabindex="-1" to ensure that programmatic focus is successfully shifted to the main content area; otherwise, the browser will merely scroll the page, and the user's next Tab keystroke will return them to the top of the document.3. User Preferences and the Media-Query SurfaceModern operating systems expose granular user preferences that browsers interpret via CSS media queries. A robust design system must map these queries to specific token layers, ensuring that components adapt natively to user needs without requiring complex JavaScript listeners.The Preference Query SurfaceThe baseline status of user preference queries in 2026 presents a mature surface for token integration:prefers-color-scheme: Widely supported, dictating the root light/dark theme variables.prefers-reduced-motion: Universally supported, serving as the system contract for animation.prefers-contrast: Widely available across major browsers, allowing systems to deliver specific WCAG enhanced contrast themes.prefers-reduced-transparency: Supported in Chromium, Edge, Safari, and Firefox as of 2024, this query targets users who experience cognitive load, legibility issues, or visual noise when confronted with blurred, translucent layering (glassmorphism).inverted-colors: This query detects if the operating system is applying a blanket color inversion matrix. However, it is fully supported only on WebKit (iOS/macOS) and currently lacks support in Chromium and Gecko, limiting its utility in cross-platform design systems.The prefers-reduced-motion contract must be applied architecturally. A design system should encode motion tokens at the foundation layer (e.g., --system-duration-fast: 200ms;). When the media query evaluates to true, these tokens automatically collapse to 0ms or a near-instantaneous transition. However, components must make local decisions regarding essential motion. For example, an indeterminate progress spinner is informational; disabling it entirely removes critical system status. Instead, the component should replace it with a static fallback or a drastically slowed animation. Conversely, parallax scrolling or layout shifts, which are known vestibular triggers, must be strictly disabled by the system when the query is active.High Contrast and Forced Colors ModeWindows High Contrast Mode (WHCM), represented in web standards as the forced-colors media query, represents a radical departure from standard rendering. To achieve this, engineers must utilize CSS System Colors via the system-color() keywords. If a design system detects prefers-color-scheme: dark and simultaneously the OS applies inversion, the dark theme is inverted back into a blindingly bright light theme, defeating the user's intent. To mitigate this double-inversion pitfall, design systems must utilize CSS filters or the inverted-colors media query (where supported) to opt specific media (like images and video) out of the inversion matrix, ensuring photographs do not appear as film negatives.4. Composite Widget Patterns and the ARIA Authoring Practices ContractDesign systems face a perpetual architectural tension between adhering to the "use the platform" philosophy and meeting complex product design requirements. Native HTML elements like <select> are highly accessible out-of-the-box, but they offer severe limitations: they cannot be deeply styled, cannot support complex multi-select operations, and cannot render rich HTML (like avatars or secondary text) within their options. When the design system must abandon the native control and implement a custom ARIA-driven widget, the implementation depth must be exhaustive.The Combobox 1.2 Paradigm and AT DisagreementsThe APG Combobox pattern is arguably the most complex composite widget to implement securely. It requires the strict coordination of an <input> element (role="combobox"), a popup mechanism (role="listbox", grid, or dialog), and child items (role="option"). The combobox must seamlessly manage aria-expanded, aria-controls, and focus delegation between the input field and the dropdown list.However, strict adherence to the APG guidance frequently conflicts with actual screen reader behavior in 2026. For example, the APG occasionally recommends utilizing a live region to announce the number of available suggestions as the user types. Yet, testing across environments reveals that appending dynamic strings to a live region during rapid keyboard input often causes screen reader buffers to flush, resulting in clipped, delayed, or chaotic announcements. Furthermore, some APG test patterns leave residual DOM content from live regions present on initial load, which screen readers incorrectly read as the first element in the document's reading order, confusing the user before they even interact with the input. The design system must architect comboboxes to debounce live region updates and selectively clear the region's text node when the combobox loses focus.The aria-owns Relationship GapIn complex grid, tree, or listbox patterns, visual design constraints often dictate layouts that prevent parent-child elements from being structurally nested within the DOM tree. To maintain accessibility, developers utilize the aria-owns attribute, which restructures the accessibility tree by programmatically forcing screen readers to treat disparate DOM nodes as a unified parent-child structure.However, rigorous testing highlights significant cross-engine support gaps for aria-owns. While JAWS on Chromium and Narrator on Edge fully support parsing multiple IDREFs via aria-owns, VoiceOver and certain Firefox configurations struggle to correctly serialize the child count. This frequently results in misnumbering the total items in a tablist or listbox (e.g., announcing "Tab 1 of 1" instead of "Tab 1 of 5"), severely disorienting the user. Therefore, the design system must enforce a strict architectural rule: aria-owns should only be utilized as a last resort. Elements should be structurally nested in the DOM wherever possible, utilizing CSS Grid or absolute positioning to achieve the desired visual layout without corrupting the native DOM hierarchy.Headless UI Library IntegrationBuilding compliant composite widgets (Combobox, Menu, Dialog, Treegrid) from scratch introduces massive technical debt. Consequently, design systems frequently evaluate headless UI libraries to serve as their functional foundation. Live Regions, Announcements, and Dynamic ContentModern web applications are highly asynchronous, continuously updating the DOM without triggering full page reloads. Sighted users visually detect these changes through layout shifts and animations, but assistive technology users remain unaware unless the system provides programmatic notifications. WAI-ARIA live regions fulfill this critical requirement.The Mechanics of aria-liveLive regions govern how and when a screen reader interrupts its current speech queue to announce a DOM mutation. The property accepts three primary values: off, polite, and assertive. A polite live region waits for the screen reader to finish its current sentence before announcing the update, making it suitable for non-critical information like "Search results loaded". An assertive region immediately flushes the speech buffer to announce critical, time-sensitive events, such as "Session expiring in 30 seconds" or "Form submission failed". Additional properties like aria-atomic dictate whether the entire region should be read upon a change, or only the specific mutated text node, while aria-relevant specifies what types of DOM mutations trigger the announcement (additions, removals, or text changes).The cardinal rule of live regions—and the most frequent point of failure in React and other SPA architectures—is that the live region container must exist in the DOM before the text node is mutated. If a component mounts a new element like <div aria-live="polite">Settings saved</div> simultaneously with its payload, the accessibility tree interprets this as a static node insertion, and the screen reader will not announce it. Design systems must architect a persistent, visually hidden live-region provider at the root of the application shell. Components should then dispatch string payload events to this central provider rather than attempting to mount isolated live regions locally.Roles and Status Messages (WCAG 2.2 SC 4.1.3)WCAG 2.2 Success Criterion 4.1.3 (Status Messages) requires that when an application provides a status update without changing the user's context (e.g., a toast notification appearing, a shopping cart counter updating, or an inline form error), the message must be programmatically determined.Design systems fulfill this by applying the role="status" attribute (which implicitly maps to aria-live="polite") for general updates, and role="alert" (mapping to aria-live="assertive") for destructive errors. Specialized roles like role="log" (for chat histories), role="timer", and role="marquee" provide more granular semantics, though engine support for specialized behavior on these roles is inconsistent.A significant challenge with SC 4.1.3 is the limitation of automated testing. While an axe-core run can statically verify that a toast component possesses role="status", automation cannot determine if the text payload was meaningfully injected at runtime, nor can it verify if the screen reader's queue actually processed and announced the message. This necessitates manual testing of the notification queue across the engine matrix to ensure rapid successions of toasts do not overwrite each other in the speech buffer.SPA Navigation and Route AnnouncementsTraditional multi-page applications trigger a hard page load upon navigation, resetting focus to the document root and prompting the screen reader to announce the new <title>. Single Page Applications (SPAs) hijack routing natively, updating the DOM via JavaScript. When a user clicks an internal link in an SPA, the URL changes and the view swaps, but the screen reader remains entirely silent, leaving the user completely disorientated.The architectural solution involves deliberate focus management. Upon a client-side route change, the SPA framework must programmatically shift focus to the primary <h1> of the new view, or to a dedicated skip link container. This action forces the screen reader to announce the heading, verifying the navigation event. If the target view lacks a focusable heading, a polite live region must be utilized to explicitly announce the route transition (e.g., "Navigated to Dashboard").6. Forms, Errors, and the Input ContractThe input layer is the primary vector for user interaction and data collection. The design system must enforce strict accessibility contracts for forms to prevent users from being stranded in unrecoverable error states.Native Forms and Validation PrimitivesWhenever possible, design systems should leverage native <form>, <label>, <input>, and <fieldset> elements. The HTML <fieldset> combined with <legend> remains the only universally robust method for logically grouping multiple inputs (such as radio button arrays or complex address blocks) and ensuring that the group description is reliably announced alongside each child input by screen readers.Modern constraint validation allows systems to leverage the CSS :user-invalid pseudo-class (achieving baseline availability across major browsers in 2024). Unlike the traditional :invalid selector, which instantly flags an empty required input as erroneous the moment the page loads (punishing the user before they even begin typing), :user-invalid respects interaction heuristics. It only applies the error state after the user has interacted with the control and fired a blur event or attempted submission. This allows the design system to seamlessly bind red validation borders and error icons without relying on complex JavaScript state tracking.The aria-errormessage and aria-describedby ParadigmWhen form validation fails, the error message text must be programmatically linked to the input. Historically, this was achieved via aria-describedby. However, aria-describedby is a blunt instrument; screen readers will announce the description regardless of whether the input is in a valid or invalid state, creating verbosity and confusion.The aria-errormessage attribute (introduced in ARIA 1.1) provides semantic precision. It points to the IDREF of the error container, but it only becomes pertinent when the input is explicitly marked with aria-invalid="true".Engine support for aria-errormessage, however, remains heavily fragmented in 2026. Testing indicates robust support in JAWS on Chromium, NVDA, and iOS VoiceOver. However, MacOS VoiceOver and Android TalkBack frequently fail to announce the referenced error payload upon focus. Because TalkBack's accessibility API requires string payloads rather than Node references for error announcements, aria-errormessage often results in silence.Therefore, until global support stabilizes, the design system should implement a hybrid approach: utilize aria-errormessage to support modern APIs, but dynamically inject the error string directly into the aria-label, or fallback to appending the error ID to the aria-describedby list when aria-invalid="true".Complex Inputs and WCAG 1.3.5For complex inputs like date pickers and drag-and-drop file uploads, native HTML has improved significantly but often falls short of enterprise requirements. While <input type="date"> is highly accessible on mobile devices (triggering native OS scrollers), its desktop implementation varies wildly across browsers and cannot easily restrict specific dates (like holidays). The design system must provide custom, ARIA-grid-based date pickers when these restrictions are necessary, ensuring full arrow-key traversal.Furthermore, WCAG 2.1 SC 1.3.5 (Identify Input Purpose) mandates the use of the autocomplete attribute on forms collecting user data. The design system must expose autocomplete properties (e.g., autocomplete="bday", autocomplete="email") so the user agent can auto-fill data, drastically reducing cognitive load and physical typing requirements for users with dexterity impairments.7. Color, Contrast, and the Visual LayerVisual accessibility is strictly quantified by WCAG mathematical thresholds. Design systems must embed these mathematical constraints directly into their color token architecture, removing the burden of manual calculation from product designers.WCAG 2.x Contrast vs. APCAThe Advanced Perceptual Contrast Algorithm (APCA) provides a highly nuanced calculation model ($L^c$) that accounts for font weight, ambient light, and spatial frequencies, successfully correcting the known flaws in the WCAG 2.x relative luminance formula (which often fails to accurately predict readability, particularly with dark mode themes and high-saturation colors).However, despite its perceptual superiority, APCA remains an exploratory technology. As of late 2023, the W3C Accessibility Guidelines Working Group officially removed APCA from the WCAG 3.0 draft due to a lack of consensus, and WCAG 3.0 itself is not expected to reach recommendation status until roughly 2030.Consequently, the legal standard for accessibility—including strict enforcement under the Americans with Disabilities Act (ADA) and the European Accessibility Act (EAA)—remains strictly bound to the WCAG 2.1/2.2 formulas. Design systems must evaluate all tokens against the WCAG 2.x math (4.5:1 for normal text, 3:1 for large text and UI components). If designers wish to utilize APCA to achieve better visual harmony across varied densities and dark modes, they must utilize Bridge-PCA formulas to ensure the resulting tokens strictly clear the legacy WCAG 2.x minimums, thereby avoiding legal compliance risk.Non-Text Contrast and Focus AppearanceWCAG 2.1 Success Criterion 1.4.11 dictates that non-text content (such as input borders, selected states, and graphical objects) must maintain a 3:1 contrast ratio against adjacent colors. To enforce this, design systems must implement a "pair-token" model, coupling foreground and background tokens via strict linting rules so that a designer cannot pair a low-contrast border with a specific background.WCAG 2.2 introduces Success Criterion 2.4.13 (Focus Appearance), evaluated at Level AAA. This criterion mandates that a focus indicator must possess a minimum surface area (at least a 2 CSS pixel thick perimeter) and maintain a 3:1 contrast ratio between its focused and unfocused states. While AAA criteria are rarely legally mandated, adopting 2.4.13 as a default specification in the design system future-proofs the focus ring architecture and significantly improves operability for sighted keyboard users.Finally, the system must enforce the rule that color is not the sole carrier of meaning. If an input relies on a red border to indicate an error, it must also include a persistent error icon or descriptive text. The design system's toolchain should provide tokens that inherently pair status colors with associated SVG icons.8. Testing, Tooling, and Conformance EvidenceA design system cannot guarantee absolute accessibility; it can only guarantee that its constituent components, when utilized correctly, will not cause inherent violations. Establishing a comprehensive automated and manual tooling pipeline is essential for generating the compliance evidence required for Voluntary Product Accessibility Templates (VPAT) and Accessibility Conformance Reports (ACR).The Limits of Automated TestingStatic analysis (via eslint-plugin-jsx-a11y) and runtime automation form the first layer of defense. Tools like Storybook's addon-a11y inject the axe-core ruleset directly into the component development environment, instantly flagging contrast failures, missing ARIA relationships, and invalid HTML structures during the build process.However, architecture leads must understand the statistical limitations of automation. Data derived from large-scale audits confirms that axe-core and similar static analysis tools can only identify an average of 57.38% of total WCAG violations. Automation cannot test semantic intent or visual context. An automated test will pass an image if it possesses alt="image of text", failing to recognize that the description is useless to the user. It can verify that an aria-errormessage ID points to an existing DOM node, but it cannot verify if the error text accurately describes how to fix the input.Browser DevTools and Manual Testing MatricesTo bridge the 42.62% testing gap, the design system must mandate manual testing across a strictly defined matrix of browsers and screen readers. The minimum reliable testing matrix must reflect the underlying rendering engines:NVDA on Chromium (Windows): The global baseline for desktop screen reader testing due to high market share.JAWS on Chromium (Windows): Required for enterprise and government compliance testing, noting its specific quirks with aria-owns and table reading modes.VoiceOver on WebKit (macOS/iOS): Critical for testing touch interactions and Apple's specific AOM translation layer.TalkBack on Chromium (Android): Necessary for mobile-specific structural evaluation and specific support gaps (like aria-errormessage).Engineers must leverage browser DevTools to inspect the underlying accessibility tree directly. Chrome DevTools provides a "Full Accessibility Tree" view, allowing developers to see the exact synthesized roles and accessible names that Blink passes to the OS. Firefox's Accessibility Inspector offers parallel capabilities but excels in highlighting JavaScript event listeners attached to specific nodes, making it invaluable for debugging custom keyboard traps and roving tabindex implementations.Conformance Evidence and VPAT GenerationFor enterprise procurement, the system must produce an ACR utilizing the standard VPAT format. Automation output (via Cypress + axe or Playwright + axe integrations running against the component library) feeds directly into the VPAT, providing verifiable proof of compliance for programmatic SCs (like 1.1.1 Non-text Content and 1.3.1 Info and Relationships). However, the manual testing matrix results provide the necessary, qualitative evidence for operational criteria, such as SC 2.1.1 Keyboard operability, SC 4.1.3 Status Messages, and SC 2.4.11 Focus Not Obscured (Minimum), which requires verifying that sticky headers do not cover focused inputs.9. The Figma Annotation Contract for WebThe primary point of failure in accessible web implementation occurs during the handoff between design and engineering. Visual designs in Figma implicitly convey hierarchy, relationship, and state through layout, proximity, and color. Because the DOM has no inherent understanding of these visual semantics, the designer must explicitly map them via a standardized annotation contract.While the agency's mobile annotation contract covers baseline properties, the web annotation contract must be substantially extended to address the unique complexities of the DOM, ARIA specifications, and the CSS media query surface.What the Designer Must AnnotateTo ensure a web engineer can implement a component without guessing semantic intent, the Figma component documentation must specify the following properties per element:Accessible Name Computation: If a component (like a pagination arrow or a hamburger menu) contains only an icon, the designer must provide the explicit string intended for the aria-label. If the label is visible on screen but visually distant from the input, the designer must explicitly draw a connecting line to denote an aria-labelledby relationship.Dynamic Relationships (IDREFs): The designer must use standardized annotation arrows to indicate structural relationships: which trigger controls which popup (aria-controls), which text node serves as the explicit help text for an input (aria-describedby), and which elements are artificially parented in complex grids (aria-owns).Modality and Inertia: For overlays, bottom sheets, and dialogs, the designer must explicitly highlight the regions of the screen that will receive the inert attribute. Critically, the designer must specify the exact focus-return target—the element that should receive keyboard focus when the modal is dismissed.Keyboard Interaction Models: For composite widgets, the designer must annotate whether the component utilizes roving tabindex or aria-activedescendant, and map the exact arrow-key traversal path according to the ARIA APG (e.g., mapping Home/End keys to jump to the start/end of a listbox).Media Query Fallbacks: The designer must define how the component behaves when forced-colors: active (e.g., mapping a transparent border to the ButtonText system color to ensure structural visibility) and when prefers-reduced-motion is active (specifying if the animation is completely halted or simplified to a crossfade).Where Defaults Render Annotations UnnecessaryTo prevent annotation fatigue and maintain efficiency, the contract must explicitly state where the web platform's defaults apply. Designers do not need to annotate:role="button" or tabindex="0" for standard design-system buttons, as the framework strictly utilizes native HTML <button> elements.Basic focus ring styling, as the system globally enforces a :focus-visible token that satisfies WCAG 2.2 SC 2.4.13.The mapping of aria-expanded on native HTML <details> and <summary> elements, as the browser platform manages this state and accessibility tree broadcast inherently.Machine-Readable Specifications and Component SchemasTo bridge the gap between static design documentation and code generation, the annotation contract can be schema-driven. By mapping the Figma annotations to structured JSON (such as the .ai.json schema or Schema.org JSON-LD formats), the design system treats components-as-data.When a designer flags a component as a "Modal Dialog" within Figma's properties, the underlying JSON payload automatically mandates an inert property for the background schema and enforces a focus-trap boolean requirement. This structured output allows the engineering code-generation pipeline to consume the schema and physically prevent the compilation of a React or Web Component that lacks the required ARIA relationships. For instance, if an input is marked as having an error state, the schema dictates the generation of useId() hooks to automatically link the input to the error text via aria-errormessage. By encoding accessibility directly into the machine-readable schema, the design system shifts accessibility from a post-development auditing task to a fundamental architectural prerequisite.
