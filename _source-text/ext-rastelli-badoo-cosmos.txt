Captured: 2026-06-18 via firecrawl_scrape

The post describes Rastelli's measurement work on Badoo's Cosmos design system. As Design System Lead at HashiCorp at time of capture; was UI Engineer at Badoo when the post was written.

The two methods Rastelli describes:

(1) UI coverage spreadsheet — the synthetic / qualitative method.
- Inventoried all UI elements composing the application; created a list of components and sub-components in a spreadsheet (~240 lines).
- Per component: status ("to do", "in progress", "to refine", "done") + a 1–5 effort estimate (story-points-shape).
- Defined a complex weighted formula comparing total points required for the entire UI vs. the points of components weighted by status (1.0 if completed, 0.8 if to refine, 0.6 if in progress, 0 if still to do).
- Split into two formulas — one optimistic, one pessimistic — producing two converging numbers over time.
- Plotted as "UI coverage over time"; trend showed growth from ~50% to ~80% in under one year; the two formulas converged (less uncertainty); approached the 80% Pareto threshold per Curtis's "average cost increases as benefit decreases" curve.
- Caveat Rastelli is explicit about: "subjective (and probably biased) metric" — based on his own perception of complexity; not objective like code-coverage.
- Useful primarily for tracking progress (rate of change), not for declaring absolute success.

(2) Source-mix-overtime via git log — the quantitative method.
- Ran `git log --stat --reverse --date=short -- 'src/css/*.less'` against the application's repo (and equivalent for Sass after the LESS-to-Sass migration).
- Wrote a Node-based JSON parser (gist linked in original post at gist.github.com/didoo/7a9fa5a95cfac32107c5358a4eba46a6) producing per-day aggregations: { date, files, insertions, deletions }.
- Produced two D3-rendered visualisations — one for the application codebase, one for the Cosmos library codebase.
- Bubbles version (radius proportional to value) — quickly showed before/after the introduction of Cosmos: large bubbles followed by small bubbles in the application codebase, correlated with the second chart for Cosmos.
- Rectangles version (linear, fixed-width with proportional height) — uncovered the secondary measure: "frequency of change" — denser rectangles = darker opacity = more frequent CSS changes.
- The chart showed before-Cosmos: many frequent medium/large changes; after-Cosmos: relatively transparent with infrequent small changes — matching Rastelli's lived experience as a UI engineer.

Rastelli's framing — explicitly *not* a metric to be reduced to a single number:
> "First, because that would be like measuring the 'number of lines committed per day' or the 'number of tickets closed per month', and this is a really bad way to measure something. When humans are involved, not everything can be measured."
>
> "Second, because I don't want to pass the idea that 'a design system makes people work less'. If you're reading this in the charts above, well let me tell you you're completely wrong. The reason for introducing a design system in a company is not because so people can work less, but because so people can work better."
