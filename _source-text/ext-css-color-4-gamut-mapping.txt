§14.2 BODY TEXT — VERBATIM (between the H3 heading and the §14.2.1 sub-heading)

> The three CSS gamut mapping algorithms apply to individual, Standard Dynamic
> Range (SDR) CSS colors which are out of gamut of an RGB display and thus
> require to be css gamut mapped.
>
> Implementations my [sic — spec typo for "may"] choose any of the three
> algorithms based on their quality and runtime efficiency tradeoffs, and must
> use their chosen algorithm wherever CSS mandates that gamut mapping be
> performed.
>
> - Binary Search Gamut Mapping with Local MINDE
> - EdgeSeeker Gamut Mapping
> - Ray Trace Gamut Mapping
>
> They all implement a relative colorimetric intent, thus colors inside the
> destination gamut are unchanged.
>
> Note: other situations, in particular mapping to printer gamuts where the
> maximum black level is significantly above zero, will require different
> algorithms which align the respective black and white points, which will
> result in lightness changes for very light and very dark colors as chroma
> is reduced.
>
> Note: these algorithms are for individual, distinct colors; for color images,
> where relationships between neighboring pixels are important and the aim is
> to preserve detail and texture, a perceptual rendering intent is more
> appropriate and in that case, colors inside the destination gamut could be
> changed.
>
> All three CSS gamut mapping algorithms aim at constant-lightness,
> constant-hue chroma reduction in the OkLCh color space.
>
> For colors which are out of range on the Lightness axis, white is returned
> in the destination color space if the Lightness is greater than or equal to
> 1.0, while black is returned in the destination color space if the Lightness
> is less than or equal to 0.0.

§14.2.1 BODY TEXT — VERBATIM (between the H4 heading and the §14.2.2 sub-heading)

> For this binary search algorithm, the color difference formula used is
> deltaEOK. The local-MINDE improvement is used. At each step in the search,
> the deltaEOK is computed between the current mapped color and a clipped
> version of that color.
>
> If the current color is outside the gamut boundary, but the deltaEOK between
> it and the clipped version is below a threshold for a just noticeable
> difference (JND), the clipped version of the color is returned as the
> mapped result. This gives good results with non-noticeable [sic — spec has
> "non-notivceable"] hue shifts, and avoids excessive chroma reduction near
> concave gamut surfaces, but can be computationally intensive.
>
> For the OkLCh color space, one JND is is [sic — spec has duplicate "is"]
> an OkLCh difference of 0.02.
>
> Note: In CIE Lab color space, where the range of the Lightness component
> is 0 to 100, using deltaE2000, one JND is 2. Because the range of Lightness
> in Oklab and OkLCh is 0 to 1, using deltaEOK, one JND is 100 times smaller.
>
> Note: for the purposes of experimentation, and comparing implementations,
> implementations of the Binary Search with Local MINDE algorithm are
> available in the Coloriade [sic — spec has "Coloriade", canonical name is
> "Coloraide"] library (in Python) and the color.js library (in JavaScript).

WHAT THE SPEC ESTABLISHES

1. **There is no single normative CSS default.** The spec presents the three
   §14.2 algorithms as siblings. "Implementations may choose any of the three
   algorithms based on their quality and runtime efficiency tradeoffs, and
   must use their chosen algorithm wherever CSS mandates that gamut mapping
   be performed." This is the load-bearing sentence. The practice's prior
   framing ("CSS Color 4 default — chroma-reduction") was conflating the
   common conceptual frame (all three are chroma-reduction) with a specific
   named algorithm. The spec does not designate one.

2. **The three algorithms share a common frame.** All three "aim at
   constant-lightness, constant-hue chroma reduction in the OkLCh color
   space" and "implement a relative colorimetric intent, thus colors inside
   the destination gamut are unchanged." The category (constant-L,
   constant-H chroma reduction in OkLCh) is the spec's framing for all
   three, not for any one default. This is what 31's body line should
   reflect.

3. **MINDE = Minimum-Delta-E.** §14.2.1 confirms the algorithm uses ΔE_OK
   (deltaEOK) as its distance metric and the local-MINDE improvement
   for handling concave gamut surfaces. One JND in OkLCh = 0.02. The
   external pass to 31's run named this "MINDE Chroma Reduction with ΔE_ok
   distancing" — the algorithm-name attribution was *partly* right (MINDE +
   ΔE_OK) but *partly* wrong (treating it as the spec's default rather than
   one of three sibling algorithms).

4. **Lightness extrema are clamped before chroma reduction runs.** L ≥ 1.0
   returns destination-space white; L ≤ 0.0 returns destination-space black.
   This applies equally to all three algorithms.

5. **Per-pixel, not per-image.** The §14.2 algorithms are explicitly for
   individual colors, not images. For images, the spec says perceptual
   rendering intent is more appropriate; the §14.2 algorithms preserve
   in-gamut colors but a perceptual remap might not.

6. **Browsers each pick one.** The spec leaves the choice to implementations.
   Which algorithm a given browser ships at a given version is an
   implementation question, not a spec question — this is what the practice
   should track if it wants to reason about what end-users actually see.

PRACTICE-FACING IMPLICATIONS

The 31 §1 gamut-mapping line previously read:

  "CSS Color 4 default (chroma-reduction) — keep L and H, reduce C until the
   color is in gamut. Default for `oklch()` colors out of gamut."

This is a **factual inaccuracy** as stated. The spec does not designate a
single algorithm as the CSS default; it specifies three siblings and lets
implementations choose. The category (constant-L, constant-H chroma reduction
in OkLCh, relative colorimetric) is the *common frame* of all three §14.2
algorithms — it is the right framing, but it should not be attributed to a
single "default" algorithm.

Recommended replacement: name the spec's frame faithfully — three §14.2
algorithms (Binary Search with Local MINDE, EdgeSeeker, Ray Trace), all aiming
at constant-L, constant-H chroma reduction in OkLCh under a relative
colorimetric intent, with implementation choice left to the browser. Note
that L ≥ 1.0 returns destination-space white and L ≤ 0.0 returns
destination-space black before the chroma-reduction step runs.

If a future engagement needs to know what a specific browser ships, the
question moves from "what does CSS Color 4 say is default" (no answer) to
"what algorithm does Chromium 149 / Firefox 151 / WebKit 26 currently
implement for `color()`/`oklch()` out-of-gamut handling" (a per-browser
implementation question with per-version answers; track via WPT results
linked from the spec — `wpt.fyi/results/css/css-color/`).
