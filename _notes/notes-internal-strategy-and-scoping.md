# Notes: AI-Compatible Design Systems & DS Scoping (Raw)

## Source meta
- **AI-Compatible Design Systems** — "Making Your Design System AI-Compatible: A strategic guide to design tokens, component metadata, agent instructions, brand extension, and evaluation frameworks for AI-assisted development." Internal POV/strategy. Feb 2025. ~57 pages. Seven chapters: Strategy Overview, Design Tokens for AI, Component Metadata & Registry, Agent Instructions, Brand Extension Playbook, Implementation Walkthrough (Button), Evaluation & Measurement.
- **DS-Scoping-Raw** — "DS Scoping." Internal practice/scoping reference. Mar 2026. 12 pages (raw, structured outline form, partly synthesized from a "WPP Open" prompt — the second half of the doc tails into a section labeled "From WPP Open" that re-frames phases). Structured as 6 numbered stages (Discovery/Research, Foundations, Components and Patterns, Documentation, Governance and Strategy, Adoption and Practice), each with Tasks + Deliverables.

## Thesis / core positioning

**Scoping doc thesis:** A design system is "a **living product** with distinct, evolving stages, rather than a static project. It aims to unify product teams, accelerate processes, and build consistency." The doc operationalizes that premise as a 6-stage tactical breakdown — Discovery, Foundations, Components/Patterns, Documentation, Governance, Adoption — each with explicit Tasks and Deliverables. It is a checklist-style scoping artifact, not an argument: it presumes the value of a DS and lays out *what to do and what to ship*.

**AI-Compatible doc thesis:** The frontier of DS practice is no longer just shipping tokens and components for human consumers — it is exposing the design system as **structured intelligence** that AI agents can consume. "AI agents are fast but brand-unaware. They know how to build interfaces, but they don't know how to build *your* interfaces." The proposed remedy is a **four-layer metadata strategy**: (1) tokens with descriptions, (2) component metadata as `.ai.json` plus a registry, (3) agent instruction files (`AGENTS.md`/`CLAUDE.md`), (4) a Figma pipeline (variables → DTCG JSON → Style Dictionary → CSS custom properties, plus Code Connect and MCP). The promise: AI-generated first drafts that are "80–90% aligned with brand standards," shifting iteration from "start over" to "refine."

Together, these documents stake out VML's current dual stance: the **scoping doc** is the operational playbook for setting up a DS as a living product; the **AI-Compatible doc** is the forward-leaning POV that says a DS done right today is one machines can read as well as humans. The Brand Extension chapter pushes farther — agents shouldn't just "use our buttons" but "build interfaces that look like our team built them," via documented composition patterns extracted from existing Figma screens and production pages.

## Phase-tagged findings

- **Discovery & Strategy** — Scoping doc, Stage 1 ("Discovery/Research"). Three task buckets: **Vision** ("Assemble the North Star" — align DS to organizational business strategy, mission, goals, served audiences, strategic priorities, vision); **Auditing** (interface audits, product+user inventory, design+code asset gathering, an **Ecosystem Diagram** showing where DS fits in the broader tech stack including CMS/CRM/DAM, identifying existing and emerging paradigms); **Stakeholder identification** (15–20 interviews with designers/engineers/PMs/writers at 30–45 min; 5–10 end-user interviews at 30 min about *broader product context, not the DS itself*; ~10 leader interviews; influence maps using green/yellow/red and arrow weight; collect feature-team roadmaps; score products against **eight factors** for pilot selection: common components/patterns, business value, technical feasibility, available champion, scope, technical independence, marketing potential — eighth factor implicit in roadmap fit). Deliverables: North Star doc, asset list, ecosystem diagram, product/user inventory, interface inventory report, paradigm identification (existing/emerging), interview notes, influence map, synthesized themes presentation, prioritized pilot list, ecosystem map of user flows.

- **Design Foundations** — Scoping doc, Stage 2. Tasks: extract **3 overarching design principles** (specific, opinionated, grown from within the team, used as decision-making tools); set KPIs at kickoff; design tokens (semantic naming with levels base/modifier/object/namespace; Style Dictionary in a JSON repo; automated delivery via GitHub Actions); style properties (color palette with primary/base + secondary/accent + use cases, typographic scale ordered by informational importance, icon library by type, standardized depth/elevation/shadows/radius, motion+sound). **Design Language Definition** uses "big levers" (scale, density, weight) and "small dials" (specific typography, spacing, color values). Deliverables: Digital Foundations document (mission, values, brand identity, tone of voice, copywriting, formatting, accessibility, responsive design, design/eng processes), 3–5 memorable principles, color/type/icon/visual-form guidelines, tokens in DTCG JSON, Style Dictionary repo, platform deliverables (CSS variables, JS modules), KPIs.

  AI doc adds the AI-compatibility layer: every token must carry a `$description` ("the highest-ROI action for AI compatibility"). Semantic naming pattern `--ds-color-action-primary-surface-rest` is "a compressed sentence" encoding namespace/category/usage/hierarchy/element/state. Semantic categories function as agent vocabulary (text, surface, border, action, feedback, disabled, space, shadow). Design for **multi-theme from day one** (light/dark/wireframe via `data-theme` attribute) so AI output is automatically theme-compatible. The `$extensions.figma` block (variableId, collection, scopes, codeSyntax) enables 1:1 Figma↔code mapping; `scopes` (e.g., `TEXT_FILL`, `STROKE_COLOR`) tell agents where a token is valid.

- **Component Library** — Scoping doc, Stage 3. **Component Hierarchy**: basic (atoms — buttons, icons), composite (molecules + organisms — combinations), containers, layout (templates + pages). **Identify 3 initial components to pilot** — frequency ≥3 across products, high business value, technical feasibility, marketing potential; buttons recommended as a starting component. Component audit grouping by *visual* and by *purpose* ("the *why* behind their existence"). **"Hot Potato" abstraction exercise** — a collaborative method to determine which characteristics are mandatory, highly recommended, or proposed for consensus. Iterate via repeated cycles. Good components are "purposeful, reusable, and flexible" — explicitly to prevent rigid single-use components that bloat the system. Pull-request workflow for contributions; usability-test new components with pilot teams.

  AI doc Layer 2 specifies what AI-compatible components require beyond a props table: each gets a `.ai.json` file with `semantic_role`, `primary_purpose`, `secondary_purposes`, `trigger_keywords` / `context_keywords` / `anti_keywords`, exhaustive `when_to_use` and `avoid_when` lists, `business_context`, `user_journey_stage`, `confidence_threshold` (0–1), `common_partners`, `required_dependencies`, `conflicts_with`, per-variant `when_to_use`, and `design_tokens_used`. Aggregated into a **component registry** (bundled CLI or CDN-hosted) for keyword search, category filter, dependency resolution, confidence ranking. Generated **component cards** (markdown + JSON, <2000 tokens each) feed RAG pipelines. JSON-schema validated in CI.

- **Documentation** — Scoping doc, Stage 4. Reference website v1 with sections: Home, Get Started, Guidelines (Accessibility, Color, Typography, Layout, Theme, Design Tokens), Component Overview, Support, Changelog. Tools considered: Storybook, Supernova, Pattern Lab, Zeroheight, GitBook, Google Docs, custom. **Usage guidelines, not just properties** — explain *why, where, when, how*; UX rationale; content guidelines. Documentation as ongoing part of design+build, "not an afterthought." Cross-disciplinary collaborative naming (designers, engineers, PMs, content strategists). Per-component minimum: name, short description, purpose, examples (design + code), do's/don'ts, accessibility rules, states (normal/hover/disabled), content guidelines, historical context. Document tokens, grid/layout/spacing, changelog/release history.

  AI doc reframes docs into a *parallel layer*: traditional human docs remain (Storybook, API.md), but a structured, machine-optimized layer (`.ai.json`, generated cards, registry manifests) sits alongside. Explicit comparison table (p.21): traditional describes how-to; metadata describes *when, why, when not to*; relationships are explicit (`common_partners`, `conflicts_with`); confidence is graduated; per-variant `when_to_use` with exact props; auto-generated, schema-validated.

- **Development Support** — Both docs. Scoping: deliver tokens to consuming repos via GitHub Actions; build technology-specific assets on top of tokens (Tailwind themes, npm packages); contribution model with criteria, review processes, "easy and rewarding"; Storybook/Git training crossing disciplines.
  AI doc Layer 3 (Agent Instructions): `AGENTS.md` (general, root) and `CLAUDE.md` (Claude Code-specific). Anatomy: project architecture context, **explicit do/don't rules** ("No hardcoded values" is "the single most important constraint"), icon/typography constraints, documentation navigation (decision tree, not flat link list), code patterns with examples, error/escalation rules ("when to ask for help"). Workflow-specific rules (e.g., Figma MCP rule file) define a **required tool flow**: `get_variable_defs` → `get_code` → `get_code_connect_map` → `get_screenshot` → translate/implement/validate. "What NOT To Do" list as hard prohibitions. Three-layer instruction architecture: project-wide / workflow-specific / component-specific. Future: MCP server exposing `list_components`, `get_component`, `search`. Practical tip: keep instruction files <600 lines; agents follow explicit prohibitions more reliably than soft guidance.

- **Pilot & Implementation** — Scoping doc Stage 3 + 5: identify 3 initial components, walkthroughs of pilot products, simple lists of modular components observed; "Conduct a Practice Pilot" before engaging real feature teams (split into "DS team" and "feature team" squads — simulate to build empathy and protocols). Component Implementation Roadmap blocks realistic timeframes per component; Pilot Team Collaboration Plans gauge interest/capacity. AI doc Chapter 6 walks Button end-to-end across all four layers — primitive tokens → semantic tokens → CSS custom properties → SCSS using only tokens (~650 lines covering all variants/states) → React TypeScript component with explicit defaults → `.ai.json` (~425 lines) → schema validation in CI → generated component card → AGENTS.md rules → Figma MCP rule flow. Provides a "chain of references" diagram (Figma variables → token JSON → CSS custom props → SCSS → React → `.ai.json` → registry → cards → AGENTS.md). **Time investment table**: token descriptions 5–10 min/category; `.ai.json` 2–4 hr/component; AGENTS.md 4–8 hr one-time; Figma MCP rules 2–4 hr one-time. For 15–20 components: ~30–80 hours of metadata authoring total.

- **Governance & Adoption** — Scoping doc Stage 5+6. Governance models named: solitary, centralized, federated, **cyclical** (combination of centralized and federated, "often recommended for its balance of ownership and contribution"). Team composition: 3–8 people, 20–30 hrs/week, roles: Product Owner, DesignOps Lead, Design Lead, Engineering Lead, Writer Lead, Designers, Engineers, PMs, Content Strategists, Accessibility Specialists, QA/Testers, Researchers. Contribution model: criteria, review process, "easy and rewarding." **"Systems Week"** — periodically dedicate one week of every three to internal-only work (breaking changes, global functionality, renames). **Versioning**: SemVer (MAJOR.MINOR.PATCH) for library and individual components. **Adoption tactics**: Practice Pilot first; shared-vocabulary glossary; join pilot sprint planning; first stakeholder update; Design System Coverage Map with statuses (Live, In Use, In Progress, In Planning); recurring "Office Hours" (semi-monthly); Slack release announcements; onboarding integration (HTML/CSS for designers, Figma for engineers, Git for designers, Storybook for designers); monitor adoption with quantitative metrics (consistency, efficiency, quality, satisfaction, reusability, maintainability, time saved, reduction in meetings/tasks/emails) and qualitative beta sentiment; support channels (Slack, office hours, Jira). Frame DS as a **collaborative product** where contribution isn't limited to code — bug fixes, feedback, suggestions all count.

- **Ongoing Maintenance** — Scoping doc Stage 5: shift from formative to operational mindset; extending, optimizing, fixing defects; delivering features; managing breaking changes; deprecating old patterns; living roadmap and backlog organized as epics/sprints/versions. AI doc Chapter 7 introduces ongoing **measurement**: periodic quarterly evaluation against a standard prompt suite; developer feedback loop (each rejected/heavily-modified AI output signals metadata, instruction, or composition gap); **metadata freshness tracking** — CI flags when a component file changes but `.ai.json` does not.

## Principles & philosophy (cross-cutting)

- **DS as a living product, not a project.** (Scoping, opening line.) This single framing carries the doc.
- **DS is a product with users.** Implicit throughout scoping (KPIs, name + logo, roadmap, marketing the release, office hours, coverage map). Echoed in AI doc's "metadata as part of the product surface."
- **Adoption is the hardest part.** Stage 6 opener: "Getting people to use the design system is often cited as the hardest part."
- **Components must be purposeful, reusable, and flexible.** Stated as a guard against rigid single-use components that bloat the system. Group components by *purpose*, not just visual similarity.
- **Principles are decision tools.** They should be "specific, opinionated, grown from within the team."
- **Big levers + small dials.** A two-tier framing of design language: big levers (scale, density, weight describe broad feel); small dials (specific typography/spacing/color values translate to granular details).
- **One source of truth, machine-traceable.** AI doc: tokens authored in Figma, exported in DTCG, processed by Style Dictionary, consumed via CSS custom properties — no translation layer where meaning gets lost.
- **Negative guidance is as important as positive.** "`avoid_when` and `anti_keywords` prevent more AI mistakes than `when_to_use` alone." (AI doc, p.22.) Hard prohibitions outperform soft suggestions for agents — this generalizes back to humans too.
- **Confidence is graduated, not binary.** A button at 0.95 vs. a PinField at lower threshold reflects how *certain* the system should be before recommending.
- **The agent should know where to look, not have everything in one file.** Decision-tree navigation > flat link lists.
- **Composition is where brand lives.** "Two login pages built with the same design system… one feels like it belongs to the brand and the other feels generic. The difference isn't in the components — it's in how they're composed." (AI doc Ch. 5.) Brand personality = layout patterns + spacing rhythms + visual hierarchy + content density + interaction models, captured *above* the component layer.
- **Failure mode for AI-aware DS:** "Many well-documented systems still produce poor AI-generated output because the documentation was written for humans, not machines."
- **Failure mode for human-only DS:** rigid single-use components, principles imposed top-down rather than grown from within, components grouped only by visual similarity, governance model unclear.

## Selling design systems (cross-cutting)

The scoping doc treats *internal selling* as ongoing work, not a one-time pitch:
- **Align DS to business strategy, mission, goals, vision** as the very first task ("Assemble the North Star").
- **Prove ROI** is a named task in Stage 5: quantify consistency, efficiency, quality, satisfaction, reusability, maintainability, time savings, reduction in meetings.
- **Influence map** — green/yellow/red, thick/thin arrows — to identify who decides and who blocks. Treat stakeholder buy-in as an explicit pre-deliverable.
- **Synthesized themes presentation** — proves stakeholders were heard; shows the DS team has a reasonable plan.
- **Regular stakeholder updates** with progress, key learnings, successes, obstacles, and clear calls to action.
- **Coverage map** as ongoing visual evidence of reach across business units (Live / In Use / In Progress / In Planning).
- **Marketing the release** — Slack posts, email digests, presentations, training events; the system is a *product* that needs comms.
- **Cross-disciplinary onboarding** as adoption fuel — teach Figma to engineers, Git to designers, etc. Adoption = literacy.

The AI-Compatible doc adds **new ROI levers** that the practice can use to sell:
- **Faster prototyping** — AI drafts start closer to final.
- **Reduced review cycles** — less back-and-forth.
- **Brand consistency at scale** — every AI-generated interface reinforces, not dilutes.
- **White-label efficiency** — same metadata strategy reskins across brand instances; tokens express identity, components stay constant.
- **Bounded investment story** — explicit time table (30–80 hours of metadata authoring for a 15–20 component system) makes the ask concrete and finite.
- **Before/After Comparison Framework** — a 5-prompt baseline test scored by rubric, then incremental layers added to show measurable impact. This is the AI doc's argument-by-experiment for funding.

## Frameworks, models, diagrams

- **Six-stage living-product lifecycle** (Scoping): Discovery/Research → Foundations → Components/Patterns → Documentation → Governance/Strategy → Adoption/Practice. Each stage has Tasks + Deliverables.
- **Eight-factor pilot scoring** (Scoping): common components/patterns, high business value, technical feasibility, available champion, scope, technical independence, marketing potential (+ roadmap fit). Used to prioritize pilot products.
- **Influence map** (Scoping): green/yellow/red node coloring × thick/thin arrows for relational influence; visual stakeholder map.
- **Ecosystem Diagram / Ecosystem Map** (Scoping): DS in tech stack (component libs, UI Kits, ref websites, guidelines, tokens, CMS/CRM/DAM); separately, user-flow map across products/platforms showing inconsistencies.
- **Component hierarchy: basic / composite / containers / layout** (Scoping): explicit four-tier layering (atoms → molecules+organisms → containers → templates+pages).
- **"Hot Potato" component abstraction** (Scoping): collaborative method to mark characteristics as mandatory, highly recommended, or proposed for consensus.
- **Big levers / small dials** (Scoping): design language framing — scale/density/weight vs. specific values.
- **Governance models**: solitary / centralized / federated / cyclical. Cyclical recommended.
- **Design System Coverage Map** with statuses Live / In Use / In Progress / In Planning.
- **Systems Week cadence**: 1 week internal work per 3 weeks pilot work.
- **Four-layer AI-compatibility stack** (AI doc): (1) Tokens with descriptions, (2) Component metadata `.ai.json` + registry, (3) Agent instruction files (`AGENTS.md`/`CLAUDE.md`), (4) Figma pipeline (variables → DTCG → Style Dictionary → CSS custom properties; Code Connect; MCP).
- **Three-layer instruction architecture**: project-wide (AGENTS.md) / workflow-specific (rule files) / component-specific (`.ai.json`).
- **Three levels of design intelligence** (AI doc Brand Extension): token-level intent / pattern-level component knowledge / interface-level composition (gestalt).
- **Required Figma MCP tool flow**: `get_variable_defs` → `get_code` → `get_code_connect_map` → `get_screenshot` → Translate→Implement→Validate.
- **Five-dimension brand-alignment rubric** (AI doc Ch. 7): Token Compliance, Component Correctness, Composition Quality, Accessibility, Design Language Fidelity. 1–5 scale per dimension; average score thresholds: 4.5–5.0 production-ready, 3.5–4.4 needs refinement, 2.5–3.4 useful start, 1.5–2.4 limited value, 1.0–1.4 infrastructure not working.
- **Expected-Impact-by-Layer table** (AI doc): hypothesizes which AI-compatibility layers produce gains in which rubric dimensions; explicitly framed as a hypothesis to be tested.
- **Token naming as compressed sentence** (AI doc): `ds-color-action-primary-surface-rest` segments → namespace, category, usage, hierarchy, element, state.
- **Chain-of-references layer diagram** (AI doc Ch. 6): Figma Variables → Token JSON → CSS Custom Properties → SCSS → React Component → `.ai.json` → Component Registry → Component Card → AGENTS.md/MCP Rules.
- **DTCG-format token export pipeline**: Figma Variables (with descriptions) → Token Forge plugin → DTCG JSON → Style Dictionary → platform outputs (CSS, SCSS, TS).

## Tensions, contradictions, open questions

- **Scoping doc fluency vs. specificity.** The scoping doc reads as an exhaustive checklist (much of it cribbable to almost any DS engagement). Some items are highly opinionated (cyclical governance recommended; buttons as first pilot; 3–8 person team at 20–30 hr/week) but most are taxonomy. It does not commit to which stages are shortenable for smaller engagements or which deliverables are mandatory vs. optional — an issue for actually scoping commercial work.
- **"Living product" vs. waterfall stages.** The doc opens with "living product" framing, then lays out a near-linear 6-stage flow. The two coexist uneasily — when does Stage 1 (Discovery) actually end if the product is living? The doc doesn't address re-entry into earlier stages.
- **AI doc vs. Scoping doc on documentation.** Scoping says docs must be "an ongoing part of the design and build process, not an afterthought" — written for humans. AI doc says "documentation was written for humans, not machines" is the failure mode. Both are right but the scoping doc has no Stage 4 deliverable that's machine-parseable. The two POVs aren't yet integrated — a senior practitioner pitching "Stage 4 + AI compatibility" has to reconcile this manually.
- **Brand Extension is partly aspirational.** AI doc Ch. 5 acknowledges interface-level patterns are "tacit knowledge — things the design team does instinctively but hasn't formalized." The Steps to Get Started (audit screens → extract patterns → validate → encode → test → iterate) are sound but heavier than they read. The "future: visual understanding" framing concedes the multimodal endpoint isn't here yet.
- **Confidence thresholds without empirical grounding.** "0.95 for Button" is asserted without explaining how a team should set or recalibrate these. The doc's Expected-Impact-by-Layer table is explicitly labeled a hypothesis.
- **"Hot Potato" is named but under-specified.** The scoping doc references it twice without describing the mechanics.
- **Overlap of governance and adoption tasks.** Stage 5 and Stage 6 share concerns (purpose statement, ROI, contribution model in 5; promotion, training, monitoring in 6). The boundary feels arbitrary.
- **AI doc end-to-end depends on Figma.** Code Connect, MCP, variable descriptions all assume Figma as design tool of record. No fallback for Sketch/Penpot/proprietary tools — limits applicability for engagements that don't standardize on Figma.
- **The "From WPP Open" handoff in scoping doc.** The doc visibly tails into a re-prompt addressed to "WPP Open" mid-Stage 6. This suggests the artifact is a working draft assembled partly from LLM dialogue, which a senior reader should know before citing it as canonical practice.

## Gaps

- **No commercial scoping translation.** Neither doc translates phases into SOW shapes, T&M ranges, or staffing curves. A practice leader pitching can't read these and produce an estimate without additional internal context.
- **No success-criteria framework for clients.** KPIs are mentioned (Stage 2) but not enumerated; ROI is mentioned (Stage 5) but not modeled. The AI doc has a rubric — the scoping doc does not have its equivalent for the *human* DS.
- **Little on multi-brand / white-label engagements.** AI doc nods to it briefly (Ch. 5 closing). Scoping doc does not address an agency's natural multi-client reality.
- **No org-readiness or maturity model.** Discovery is detailed, but there's no maturity framework (e.g., InVision-style 4-stage, Sparkbox, or VML-internal) to place a client on a curve and prescribe stages by maturity.
- **Tooling opinions are thin.** Scoping doc lists Storybook/Supernova/Pattern Lab/Zeroheight/GitBook without trade-offs. AI doc is opinionated about Style Dictionary, DTCG, MCP — but doesn't address competing token tooling (Tokens Studio, Specify, Knapsack).
- **No accessibility-first stance.** Accessibility appears as a checklist item in both docs but isn't elevated as a principle. Given VML's consultancy positioning this is a notable absence.
- **No discussion of design ops staffing patterns** beyond the 3–8 / 20–30hr line. No skill-mix breakdown by stage, no contractor vs. embedded recommendation.
- **No risk register or anti-patterns catalog.** What kills DS engagements? The docs are uniformly aspirational; neither catalogs failure modes from prior work.
- **Pricing/contracting models absent.** Subscription? Retainer? Project? Federated chargeback? Untouched.
- **Education and enablement** beyond onboarding tasks (HTML/CSS for designers, etc.) is sparse — no curriculum, no certification path, no "DS literacy" frame.
- **Sustainability / deprecation lifecycle.** Briefly named ("deprecating old patterns") but not modeled — how does a component leave the system gracefully?
- **AI doc skips human-in-the-loop governance.** No discussion of how design review of AI output integrates with the contribution model, or how human approvers escalate metadata gaps back into the system.
